### Idea 1: Product Mode 분기 로직 — Strategy 패턴 기반 플러그인화

**발견 근거**:
- `feat: product mode toggle`, `feat: domain-specific analysis focus areas`, `feat: general mode UI` 3개 커밋이 5일 이내에 연속 추가되어 mode 분기가 급격히 팽창
- 핫스팟 파일 `lib/prompts.ts`(4회), `lib/claude.ts`(4회), `InputSection.tsx`(4회)에 mode 조건 분기가 혼재될 가능성이 높음
- 과거 JSON 파싱 버그의 재발 패턴(조건 분기 → 엣지 케이스 누락 → fix 커밋 반복)이 mode 로직에서 재현될 위험

**현재 문제**:
- `lib/prompts.ts` 내부에 `if (mode === 'yafit') { ... } else { ... }` 형태의 분기가 늘어날수록 신규 도메인 추가 시 기존 로직 전체를 수정해야 하는 산탄총 수술(Shotgun Surgery) 발생
- `InputSection.tsx`(824줄 이미 과대)에 mode-aware UI 분기가 추가되면 컴포넌트 책임이 더욱 확산
- 테스트 시 mode 조합 폭발(분석 타입 × mode × 도메인)로 QA 커버리지 확보 불가

**제안**:
```
lib/modes/
├── index.ts              ← ModeStrategy 인터페이스 정의
├── yafitMode.ts          ← 야핏무브 전용 프롬프트·관점·도메인 설정
├── generalMode.ts        ← 일반 서비스 전용 설정
└── modeRegistry.ts       ← 모드 등록 및 런타임 조회

# 변경 방향
- lib/prompts.ts → 모드 중립 템플릿만 보유, 도메인 설정은 mode 모듈에서 주입
- InputSection.tsx → useModeConfig() 훅으로 UI 분기 추상화
- app/api/analyze/route.ts → ModeStrategy.getAnalysisConfig()로 단일 진입점 유지
```
신규 도메인(예: 커머스, 헬스케어) 추가 시 단일 파일만 추가하면 되도록 개방-폐쇄 원칙 적용.

**구현 난이도**: Medium
**예상 임팩트**: High
**영향 파일**:
- `lib/prompts.ts`
- `lib/claude.ts`
- `components/InputSection.tsx`
- `app/api/analyze/route.ts`
- `app/api/analyze/handlers/types.ts`

### Idea 2: ImprovementPanel 상태 머신 도입 — reanalyze 버그 재발 방지

**발견 근거**:
- `fix(improvement): fix reanalyze button doing nothing` → 5일 후 `fix(improvement): resolve race condition causing reanalyze button to do nothing` 동일 버그 **2회 재발**
- `ImprovementPanel.tsx`가 핫스팟 파일 2위(5회 변경)이며, 같은 기간 opusGenerator(3회), generate-improvement API(1회)까지 개선 관련 파일이 집중적으로 변경
- variant count 선택, PNG 저장, 3-variant 지원, screenshot panel 전환 등 상태 조합이 급증했으나 상태 관리가 ad-hoc 방식으로 추가되는 중

**현재 문제**:
- reanalyze 버튼의 race condition은 "버튼 클릭 → API 호출 중 상태 → 완료/실패 복귀"라는 비동기 전환을 명시적 상태 머신 없이 `useState` 플래그 조합으로 관리하기 때문에 발생
- variant(1~3개) × 생성 진행 단계 × screenshot 패널 표시 여부 조합이 증가할수록 플래그 동기화 오류 확률 상승
- 현재 구조에서는 같은 버그가 새로운 기능 추가 때마다 재출현할 가능성이 높음

**제안**:
```typescript
// lib/improvement/stateMachine.ts
type ImprovementState =
  | { status: 'idle' }
  | { status: 'generating'; variantCount: number; progress: number }
  | { status: 'success'; variants: Variant[]; selectedIndex: number }
  | { status: 'error'; reason: string; retryable: boolean };

type ImprovementEvent =
  | { type: 'GENERATE'; variantCount: number }
  | { type: 'PROGRESS'; value: number }
  | { type: 'SUCCESS'; variants: Variant[] }
  | { type: 'FAIL'; reason: string }
  | { type: 'REANALYZE' }
  | { type: 'RESET' };

// useImprovementMachine() 훅으로 ImprovementPanel에 주입
// 상태 전환 규칙을 순수 함수(reducer)로 정의 → 단위 테스트 용이
```
XState 또는 자체 reducer 패턴으로 구현. reanalyze는 `REANALYZE` 이벤트 → `idle` 전환 후 재시작으로 race condition 구조적 제거.

**구현 난이도**: Medium
**예상 임팩트**: High
**영향 파일**:
- `components/ImprovementPanel.tsx`
- `lib/improvement/opusGenerator.ts`
- `app/api/generate-improvement/route.ts`
**예상 작업량**: 2~3일

### Idea 3: 개선안 수락/거절 피드백 루프 — AI 제안 품질 자기강화 시스템

**발견 근거**:
- `rewrite generator prompt for surgical fixes, not full redesign` 커밋: AI가 전면 리디자인을 제안하는 문제를 프롬프트 재작성으로 수동 수정 — 사용자 피드백이 없어 AI 품질을 개발자가 직접 조정하는 구조
- `인사이트 대시보드 + AI 역제안 기능` 및 `score interpretation, benchmarks, trend context` 커밋: 대시보드가 분석 결과를 집계하는 방향으로 진화 중이나, **개선안에 대한 사용자 반응 데이터는 수집 안 됨**
- `app/api/dashboard/insights/route.ts`가 핫스팟(3회): 인사이트 계층이 이미 확장 중이므로 피드백 데이터 추가 비용 낮음

**현재 문제**:
- 사용자가 3개 variant 중 어떤 것을 채택하거나 무시했는지 기록되지 않아, 어떤 유형의 AI 제안이 실제로 유용한지 알 수 없음
- 프롬프트 품질 개선이 개발자의 주관적 판단에만 의존 → 동일한 "너무 급진적인 제안" 문제가 반복 출현할 위험
- 히스토리 페이지에 재분석 기능은 있지만, 이전 개선안과 현재 개선안의 차이를 비교하는 UI가 없어 개선 사이클의 가치가 낮음

**제안**:
```
# 1단계: 경량 피드백 수집 (Low effort)
ImprovementPanel variant 카드에 👍/👎 + "적용함" 버튼 추가
→ localStorage에 { analysisId, variantIndex, action, timestamp } 저장

# 2단계: 인사이트 대시보드 연동 (Medium effort)
app/api/dashboard/insights/route.ts에 improvement_feedback 집계 추가
→ "수락률 높은 개선 유형", "자주 거절된 패턴" 카드 노출

# 3단계: 프롬프트 자기강화 (Medium effort)
lib/improvement/opusGenerator.ts에서 과거 수락 패턴을 
few-shot 예시로 프롬프트에 자동 주입
→ 도메인별, 심각도별 "잘 먹히는 제안 스타일" 학습
```
1단계만 구현해도 사용자에게 "내 결정이 기록된다"는 인식을 주어 재방문율 향상 기대.

**구현 난이도**: Low (1단계) → Medium (2~3단계)
**예상 임팩트**: High
**영향 파일**:
- `components/ImprovementPanel.tsx`
- `app/api/dashboard/insights/route.ts`
- `lib/improvement/opusGenerator.ts`
- `app/history/page.tsx`
**예상 작업량**: 1단계 1일 / 전체 4~5일
