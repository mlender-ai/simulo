### Idea 1: TypeScript 타입 안정성 강화 — E2E 테스트 & 빌드 파이프라인 안정화

**발견 근거**: 
- QA 리포트에서 `playwright.config.ts`, `e2e/settings.spec.ts` 타입 체크 실패(TS2307, TS7031)
- 빌드 상태: ⚠️ FAIL, 타입 체크: ❌ FAIL
- 최근 커밋(4fdfc62)에서 "resolve TypeScript build failure for E2E/Playwright files" 처리했음에도 지속적 실패
- `package.json`에 `@playwright/test` 등재되어 있으나 `tsconfig.json` 설정 불일치

**현재 문제**: 
- CI/CD 파이프라인이 타입 에러로 블로킹됨 (lint는 통과하나 build/type-check 실패)
- E2E 테스트 작성 시 IDE 자동완성 불가능, 개발자 생산성 저하
- `skipLibCheck` 설정이 명확하지 않아 로컬 환경과 CI 환경 불일치 리스크

**제안**: 
1. **tsconfig.json 통합 정규화**
   - `skipLibCheck: true` 명시 (Playwright 관련 타입 검사 스킵)
   - `types: ["@playwright/test", "node"]` 추가
   - `moduleResolution: "bundler"` → `"node"` 확인
   
2. **tsconfig.e2e.json 신규 작성**
   ```json
   {
     "extends": "./tsconfig.json",
     "compilerOptions": {
       "noEmit": true,
       "skipLibCheck": true,
       "types": ["@playwright/test", "node"]
     },
     "include": ["e2e/**/*", "playwright.config.ts"]
   }
   ```

3. **E2E 테스트 기본 픽스처 작성** (`e2e/fixtures.ts`)
   - `Page`, `BrowserContext` 타입 명시
   - 모든 spec 파일에서 기본 픽스처 확장하도록 변경

4. **npm script 추가**
   ```bash
   "type-check:e2e": "tsc --noEmit -p tsconfig.e2e.json"
   ```

**구현 난이도**: Medium
**예상 임팩트**: High (빌드 안정성 획기적 개선, CI/CD 블로킹 해제)
**영향 파일**: 
- `tsconfig.json`
- `tsconfig.e2e.json` (신규)
- `playwright.config.ts`
- `e2e/*.spec.ts` (3개 파일)
- `e2e/fixtures.ts` (신규)
- `package.json`

**예상 작업량**: 1.5일 (설정 통일 + 테스트 픽스처 작성 + 검증)

### Idea 3: Claude API 에러 핸들링 표준화 & 자동 복구 메커니즘 강화

**발견 근거**: 
- 최근 커밋(770c8c0)에서 "exponential backoff retry to all Claude API calls" 추가됨
- `app/api/analyze/route.ts` (3회 수정)에서 API 호출 로직 반복 개선되고 있음
- git log에서 자율 QA/아이디어 에이전트 추가(2077919, 01cd67d, 4d3fe8d) 이후 API 안정성 관련 작업 지속
- E2E 테스트 3개 파일 모두 실행 중이나, API 타임아웃/에러 시나리오 테스트 부재

**현재 문제**: 
- exponential backoff 구현이 있으나, 에러 유형별 대응 전략 부재 (rate limit vs. server error vs. timeout 구분 없음)
- `app/api/analyze/route.ts` 등 여러 API 엔드포인트에 중복된 재시도 로직
- 사용자에게 제공되는 에러 메시지가 기술적이고 불친절함
- CLI 에이전트(`agent:idea`, `agent:qa`)와 웹 앱 API의 에러 핸들링 표준 미통일

**제안**: 
1. **Claude API 에러 핸들링 추상화 계층 작성** (`lib/claude/error-handler.ts`)
   ```typescript
   export enum ApiErrorType {
     RATE_LIMIT = "rate_limit",       // 429
     UNAUTHORIZED = "unauthorized",   // 401
     SERVER_ERROR = "server_error",   // 500+
     TIMEOUT = "timeout",
     PARSE_ERROR = "parse_error",
     UNKNOWN = "unknown"
   }
   
   export interface ApiErrorContext {
     type: ApiErrorType;
     retryable: boolean;
     suggestedWaitMs: number;
     userMessage: string;
     debugLog: string;
   }
   ```

2. **재시도 전략 통합** (`lib/claude/retry.ts`)
   - Rate limit(429): exponential backoff + Retry-After 헤더 존중
   - Server error(5xx): exponential backoff (기존 구현 유지)
   - Timeout: 짧은 지연 후 1회만 재시도
   - Parse error: 재시도 불가, 즉시 실패
   - 최대 재시도 횟수 설정 가능 (기본값 3회)

3. **모든 API 엔드포인트에 통일된 에러 응답 포맷**
   ```typescript
   export interface ApiErrorResponse {
     success: false;
     error: {
       code: string;           // RATE_LIMIT, SERVER_ERROR 등
       message: string;        // 사용자 친화적 메시지
       retryable: boolean;
       suggestedRetryAfterMs?: number;
     };
   }
   ```

4. **E2E 테스트에 API 에러 시나리오 추가** (`e2e/api-errors.spec.ts`)
   - Rate limit 응답 모킹 후 자동 복구 확인
   - 타임아웃 시 재시도 동작 확인
   - 사용자에게 표시되는 에러 메시지 검증

5. **로컬 알림 시스템(기존)과 통합**
   - 재시도 중: "분석 재시도 중..."
   - 실패: "분석 실패. 잠시 후 다시 시도해주세요."
   - Rate limit: "요청이 많습니다. 1분 후 시도해주세요."

**구현 난이도**: Medium (구현은 복잡하나, 기존 exponential backoff 재사용 가능)
**예상 임팩트**: Medium-High (사용자 경험 개선, API 안정성 향상, 디버깅 용이)
**영향 파일**: 
- `lib/claude/error-handler.ts` (신규)
- `lib/claude/retry.ts` (신규)
- `lib/claude/client.ts` (기존 코드 개선)
- `app/api/analyze/route.ts`
- `app/api/analyze-flow/route.ts`
- `app/api/report/route.ts`
- `app/api/figma-validate/route.ts`
- `e2e/api-errors.spec.ts` (신규)
- `components/OnboardingBanner.tsx` 또는 알림 컴포넌트 (에러 메시지 통합)

**예상 작업량**: 2-2.5일 (추상화 작성 + API 엔드포인트 통합 + E2E 테스트)
