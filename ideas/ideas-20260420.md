### Idea 1: Claude API 재시도 로직의 명시적 상태 추적 시스템 도입

**발견 근거**: 
- 최근 커밋 `770c8c0`에서 "exponential backoff retry to all Claude API calls" 추가됨
- 이후 `8187613` ~ `9670dc9` (4일) 동안 매일 자동 아이디어 리포트가 반복 생성됨
- `lib/claude.ts`가 4회 수정되었으나 근본 원인 미파악 상태

**현재 문제**: 
- Backoff 재시도가 성공/실패/무한 루프 중 어느 상태인지 추적 불가능
- 응답 파서(`lib/response-parser.ts`)가 부분 응답을 처리할 때 재시도 트리거 상태 불명확
- 에러 로그가 `.claude/memory/errors.md`에 기록되지 않음 (파일 미생성)

**제안**: 
1. `lib/claude/retry-tracker.ts` 신규 모듈 추가
   - 각 API 호출마다 `{requestId, timestamp, attempt, status, lastError}` 추적
   - IndexedDB에 마지막 100개 요청 이력 저장 (기존 이미지 저장처럼)
2. `app/api/analyze/route.ts`에서 실패 응답 시 `x-retry-count`, `x-last-error` 헤더 반환
3. `.claude/memory/errors.md` 자동 생성 로직 추가 (매시간 갱신, 최근 24시간 에러만 유지)
4. 디버그 대시보드: `/api/health` 엔드포인트에 최근 5개 실패 요청 정보 추가

**구현 난이도**: Medium
**예상 임팩트**: High (API 안정성 문제의 근본 원인 파악 가능 + 재발 방지)
**영향 파일**: 
- `lib/claude.ts` (기존)
- `lib/claude/retry-tracker.ts` (신규)
- `app/api/analyze/route.ts` (수정)
- `app/api/health/route.ts` (확장)
- `.claude/memory/errors.md` (자동 생성)

**예상 작업량**: 6-8시간

### Idea 2: 대용량 핫스팟 파일의 계층적 리팩토링 (점진적 모듈 분리)

**발견 근거**: 
- `lib/claude.ts` (893줄) → 4회 수정
- `components/InputSection.tsx` (824줄) → 3회 수정  
- `components/ReportTabs.tsx` (811줄) → 포함된 수정 이력 미추적
- 최신 QA 리포트에서 "대용량 컴포넌트 파일(823~893줄)"로 명시된 SRP 위반 지적
- TypeScript 빌드 실패의 일부 원인이 복잡한 파일 구조에서의 타입 추론 어려움

**현재 문제**: 
- 단일 파일에서 API 통신 + 프롬프트 생성 + 응답 파싱 + 캐싱을 모두 처리 (claude.ts)
- InputSection이 파일 업로드 + 파라미터 제어 + 프리셋 선택을 모두 포함 → 테스트 작성 시 전체 로직을 mock해야 함
- 변경 시 의도치 않은 부작용 증가 (예: response-parser 수정이 claude.ts에 영향)

**제안**: 
1. **Phase 1** (1주): `lib/claude.ts` 분리
   ```
   lib/claude/
   ├── index.ts (기존 claude.ts의 public API만 export)
   ├── client.ts (Anthropic SDK 래핑 + 재시도 로직)
   ├── prompt.ts (프롬프트 템플릿 + 빌더)
   └── cache.ts (응답 캐싱 전용)
   ```
2. **Phase 2** (1주): `components/InputSection.tsx` 분리
   ```
   components/input/
   ├── InputSection.tsx (컨테이너)
   ├── FileUploadArea.tsx
   ├── ParameterPanel.tsx
   └── hooks/useInputState.ts
   ```
3. 각 분리 단계마다 단위 테스트 작성 (jest)
4. 분리 후 번들 사이즈 측정 및 lazy loading 기회 발굴

**구현 난이도**: High
**예상 임팩트**: High (코드 유지보수성 +30~40%, 테스트 커버리지 확대 가능, 변경 리스크 감소)
**영향 파일**: 
- `lib/claude.ts` → `lib/claude/*` (분리)
- `lib/response-parser.ts` (독립도 증가)
- `components/InputSection.tsx` → `components/input/*` (분리)
- `app/page.tsx` (import 경로 변경)
- `app/api/analyze/route.ts` (import 경로 변경)

**예상 작업량**: 10-14일 (2주 스프린트)

### Idea 3: TypeScript 컴파일 파이프라인 표준화 및 E2E 테스트 타입 안정성 강화

**발견 근거**: 
- 최신 QA 리포트 (2026-04-15): 빌드 ⚠️FAIL, 타입 체크 ❌FAIL
- 근본 원인: `@playwright/test` 모듈 미등재 + `tsconfig.json` vs `playwright.config.ts` 설정 불일치
- E2E 테스트 3개 파일 모두 암묵적 `any` 타입 (TS7031) → 런타임 에러 위험
- `package-lock.json` 584줄 증가 → 의존성 관리 복잡화 중

**현재 문제**: 
- CI/CD에서 빌드 실패하지만 로컬 개발 서버는 정상 실행 → 환경 간 불일치
- E2E 테스트는 자동으로 실행되지만 타입 검증 없음 → 리팩토링 시 숨겨진 버그 위험
- `npm run type-check:e2e`는 존재하지만 CI 파이프라인에 미통합

**제안**: 
1. 공통 TypeScript 설정 통합
   ```json
   // tsconfig.base.json (신규)
   {
     "compilerOptions": {
       "skipLibCheck": true,
       "moduleResolution": "node",
       "types": ["@playwright/test", "node", "jest"]
     }
   }
   
   // tsconfig.json, tsconfig.e2e.json이 extends
   ```

2. `playwright.config.ts`에서 tsconfig 명시적 참조
   ```typescript
   export default defineConfig({
     use: { ...devices['Desktop Chrome'] },
   });
   ```

3. GitHub Actions 워크플로우 개선 (`.github/workflows/validate.yml`)
   ```yaml
   - run: npm run lint
   - run: npm run type-check
   - run: npm run type-check:e2e  # ← 신규 추가
   - run: npm run test
   - run: npm run build
   ```

4. E2E 테스트 기본 fixture 생성 (타입 안전성 확보)
   ```typescript
   // e2e/fixtures.ts
   import { test as base, Page } from '@playwright/test';
   export const test = base.extend<{ authenticatedPage: Page }>({...});
   ```

5. `e2e/*.spec.ts` 모두 `fixtures.ts`에서 import

**구현 난이도**: Low
**예상 임팩트**: High (CI/CD 신뢰도 +100%, 버그 조기 발견율 +40~60%, 개발자 경험 개선)
**영향 파일**: 
- `tsconfig.json` (수정)
- `tsconfig.e2e.json` (신규 또는 수정)
- `tsconfig.base.json` (신규)
- `playwright.config.ts` (수정)
- `e2e/*.spec.ts` (타입 선언 추가)
- `e2e/fixtures.ts` (신규)
- `.github/workflows/validate.yml` (또는 `qa-agent.ts`)
- `package.json` (스크립트 검증)

**예상 작업량**: 4-6시간
