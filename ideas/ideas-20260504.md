### Idea 1: 개선안 생성 파이프라인 상태 머신 도입

**발견 근거**:
- `fix(improvement): resolve race condition causing reanalyze button to do nothing` 커밋(41a89db)이 14일 이내 동일 증상으로 2회 등장 (fd360ff → 41a89db)
- 핫스팟 상위 3개 파일(`ImprovementPanel.tsx` 5회, `opusGenerator.ts` 3회, `lib/claude.ts` 3회)이 모두 동일한 개선안 생성 흐름에 집중
- race condition이 반복된다는 것은 컴포넌트 내부 `useState` + 이벤트 핸들러로 비동기 흐름을 제어하는 현재 구조가 근본적으로 취약함을 의미

**현재 문제**:
- "재분석" 버튼 클릭 시 응답 없음 증상이 동일 파일에서 2주 내 재발 → 상태 전이 로직이 컴포넌트 안에 분산되어 있어 어떤 조건에서 버튼이 활성/비활성인지 추론 불가능
- `ImprovementPanel.tsx`가 생성 요청, 로딩, 결과 표시, 에러 처리, 재시도를 모두 직접 관리하여 각 상태 전이 시 side effect 충돌 발생
- `opusGenerator.ts`와 `lib/claude.ts` 사이의 호출 계약이 명확하지 않아 응답 지연/중단 시 UI 상태가 stale하게 남는 현상

**제안**:
XState 또는 경량 상태 머신 패턴(`useReducer` 기반)으로 개선안 생성 흐름을 명시적 상태로 관리

```
idle → requesting → streaming → success
                 ↘ error → idle (재시도 가능)
```

- `lib/improvement/stateMachine.ts` 신규 생성: `idle | requesting | streaming | success | error` 5개 상태와 전이 조건을 단일 파일에 선언
- `ImprovementPanel.tsx`는 상태 머신의 현재 state를 구독하여 렌더링만 담당 (비동기 로직 제거)
- `opusGenerator.ts`는 순수 함수로 분리하여 상태 전이 트리거 역할만 수행
- 재분석 버튼은 `state === 'idle' || state === 'error'`일 때만 활성화되도록 단일 조건으로 통제

**구현 난이도**: Medium
**예상 임팩트**: High
**영향 파일**:
- `components/ImprovementPanel.tsx`
- `lib/improvement/opusGenerator.ts`
- `lib/claude.ts`
- `lib/improvement/stateMachine.ts` (신규)

**예상 작업량**: 2~3일

### Idea 2: InputSection 입력 파라미터 스키마 기반 동적 렌더링 구조로 전환

**발견 근거**:
- `InputSection.tsx`가 4회 수정된 핫스팟이며 QA 리포트 기준 이미  824줄
- 최근 7일 동안 description 입력, reference image 입력, multi-persona 옵션, 배리언트 수 선택, analysis preset save/load, domain selector, product mode toggle 등 **7개의 독립적 입력 파라미터**가 연속으로 추가됨
- 각 파라미터가 별도 `useState`로 관리되고 있어 파라미터 추가 시마다 `InputSection.tsx` 전체를 수정해야 하는 구조적 패턴이 고착

**현재 문제**:
- 신규 입력 파라미터가 추가될 때마다 `InputSection.tsx` 한 파일이 수정되어 다른 입력 필드의 동작에 의도치 않은 영향을 줄 위험이 선형으로 증가
- 824줄 단일 파일은 각 입력 필드의 show/hide 조건, validation, 기본값이 뒤섞여 있어 "어떤 모드에서 어떤 파라미터가 노출되는가"를 추론하기 어려움
- product mode toggle(야핏무브 ↔ 일반), domain selector, analysis preset이 각각 독립적으로 구현되어 있으나 세 기능이 모두 "분석 컨텍스트 설정"이라는 동일 목적을 가짐 → 사용자에게 파라미터 설정이 파편적으로 노출됨

**제안**:
입력 파라미터를 선언적 스키마로 정의하고, 스키마 기반으로 UI를 동적 렌더링하는 구조로 전환

```typescript
// lib/input/inputSchema.ts (신규)
export const INPUT_SCHEMA: InputParam[] = [
  {
    id: 'mode',
    type: 'toggle',
    label: '분석 모드',
    showWhen: () => true,
  },
  {
    id: 'description',
    type: 'textarea',
    label: '화면 설명',
    showWhen: (ctx) => ctx.mode !== 'comparison',
  },
  {
    id: 'referenceImage',
    type: 'file',
    label: '참고 이미지',
    showWhen: (ctx) => ctx.improvement === true,
  },
  // ...
]
```

- `InputSection.tsx` → 스키마를 순회하여 렌더링하는 컨테이너로 축소 (~200줄 목표)
- 각 입력 타입별 렌더러(`TextareaInput`, `FileInput`, `ToggleInput`, `SelectInput`)를 `components/input/` 하위에 분리
- `showWhen` 조건을 스키마에 명시함으로써 "어떤 파라미터가 언제 보이는가"를 코드가 아닌 데이터로 문서화
- 신규 파라미터 추가 시 스키마에 항목 1개 추가로 완결 → `InputSection.tsx` 직접 수정 불필요

**구현 난이도**: Medium
**예상 임팩트**: Medium
**영향 파일**:
- `components/InputSection.tsx`
- `lib/input/inputSchema.ts` (신규)
- `components/input/TextareaInput.tsx` (신규)
- `components/input/FileInput.tsx` (신규)
- `components/input/ToggleInput.tsx` (신규)
- `components/input/hooks/useInputState.ts` (신규)

**예상 작업량**: 2~3일

### Idea 3: TypeScript 빌드 파이프라인 복구 및 E2E 타입 계약 자동 검증 CI 게이트 추가

**발견 근거**:
- QA 리포트(2026-04-15): 타입 체크 ❌ FAIL, 빌드 ⚠️ FAIL 상태가 현재 커밋 기준으로도 미해소
- `app/api/analyze/handlers/types.ts`가 3회 수정된 핫스팟 — 핸들러 간 타입 계약이 불안정하게 변경되고 있음에도 타입 체크가 CI에서 통과 기준이 아님
- `feat: add minimum data guard and typed stats contract for insights`(93526e0) 커밋에서 볼 수 있듯이 타입 계약 강화 작업이 개별 기능 단위로 파편적으로 이루어지고 있어 전체 빌드 유효성이 보장되지 않음
- `fix(improvement): restore panel, add 3-variant support + PNG save; improve heatmap accuracy via OCR coords`처럼 하나의 fix 커밋에 기능 복구 + 신규 기능이 혼재하는 패턴은 타입 불안정 상태에서 빠른 배포를 위해 타입 검사를 우회하는 압력이 존재함을 시사

**현재 문제**:
- `@playwright/test` 모듈 미등록으로 E2E 테스트 3개 파일 전체에서 TS2307 에러 발생 → 타입 체크가 CI에서 blocking 조건이 아니므로 에러가 수 주째 방치 가능한 상태
- `app/api/analyze/handlers/types.ts`의 타입 변경이 실제 API 핸들러(`url.ts`, `figma.ts` 등)에 전파되는지 자동으로 검증하는 단계 없음 → 핸들러 간 계약 drift 발생
- 빌드 FAIL 상태에서 Vercel 배포가 진행되는 구조라면 프로덕션 안정성이 CI와 분리된 상태

**제안**:
TypeScript 빌드를 CI의 hard blocking gate로 복구하고, 핸들러 타입 계약 검증을 자동화

1. **즉시 수정 (1일)**
   ```bash
   npm install --save-dev @playwright/test@latest @types/node
   ```
   `tsconfig.json`에 `types: ["@playwright/test", "node"]` 추가
   `e2e/` 디렉토리 전용 `tsconfig.e2e.json` 분리하여 메인 빌드와 충돌 방지

2. **핸들러 타입 계약 자동 검증 (1~2일)**
   ```typescript
   // app/api/analyze/handlers/__tests__/types.contract.test.ts (신규)
   // 각 핸들러의 input/output이 types.ts 스키마와 일치하는지 검증
   // 기존 CI schema contract tests(04f1545)와 통합
   ```
   `app/api/analyze/outputSchema.ts`를 기준 스키마로 사용하여 각 핸들러 응답을 `zod` 또는 `@effect/schema`로 런타임 검증

3. **GitHub Actions / Vercel CI 게이트 추가**
   ```yaml
   # .github/workflows/ci.yml
   - run: npm run type-check   # tsc --noEmit (blocking)
   - run: npm run build        # blocking
   - run: npm run test:e2e     # non-blocking (별도 job)
   ```
   타입 체크 실패 시 Vercel Preview 배포 차단

**구현 난이도**: Low
**예상 임팩트**: High
**영향 파일**:
- `playwright.config.ts`
- `e2e/settings.spec.ts`
- `tsconfig.json`
- `tsconfig.e2e.json` (신규)
- `app/api/analyze/handlers/types.ts`
- `app/api/analyze/outputSchema.ts`
- `app/api/analyze/handlers/__tests__/types.contract.test.ts` (신규)
- `.github/workflows/ci.yml` (신규)

**예상 작업량**: 1~2일
