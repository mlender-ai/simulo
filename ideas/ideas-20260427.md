### Idea 1: 입력 모드별 컨텍스트 어댑터 레이어 도입 — InputSection 분기 복잡도 해소

**발견 근거**:
- `components/InputSection.tsx`가 핫스팟 Top 2 (4회 변경), 824줄로 SRP 위반 상태
- 최근 7일 내 URL 라이브 분석(`30cde33`), OCR 파이프라인(`2400afd`), 영상 업로드(`39781f4`), Figma 역방향 분석(`18f1802`) 등 4가지 입력 모드가 연속 추가
- QA 리포트: `InputSection.tsx` 분리 권장 (HIGH 우선순위)
- `app/page.tsx`도 5회 변경으로 핫스팟 Top 1 — 입력 모드 추가마다 최상단 페이지까지 파급되는 패턴 반복 확인

**현재 문제**:
- 이미지·URL·영상·Figma·비교 등 5개 이상의 입력 모드가 단일 컴포넌트 내 `if/switch` 분기로 처리되어, 신규 모드 추가 시 기존 로직 부작용 위험 상존
- 입력 모드별 유효성 검사·미리보기·에러 메시지 로직이 뒤섞여 버그 재발 시 원인 파악 비용 증가
- 실제로 `7b9cded`(PNG 업로드 에러), `90d6357`(필수 항목 에러 UI)가 모두 InputSection 영역에서 발생

**제안**:
각 입력 모드를 독립된 **Input Adapter** 단위로 분리하고, `InputSection`은 어댑터를 조합하는 컨테이너로만 남긴다.

```
components/input/
├── InputSection.tsx          ← 어댑터 등록·선택 컨테이너만 담당
├── adapters/
│   ├── ImageInputAdapter.tsx   ← 파일 드래그앤드롭, 미리보기
│   ├── UrlInputAdapter.tsx     ← URL 입력, 라이브 스크린샷 트리거
│   ├── VideoInputAdapter.tsx   ← 영상 업로드, 프레임 선택
│   ├── FigmaInputAdapter.tsx   ← Figma URL, 토큰 처리
│   └── ComparisonAdapter.tsx   ← Before/After 쌍 입력
├── hooks/
│   ├── useInputMode.ts         ← 활성 모드 상태 관리
│   └── useInputValidation.ts   ← 모드별 유효성 검사 통합
└── types/
    └── InputAdapterTypes.ts    ← 공통 인터페이스 정의
```

각 어댑터는 `{ value, isValid, errorMessage, preview }` 공통 인터페이스를 구현하여, `InputSection`이 모드와 무관하게 동일 방식으로 처리 가능하게 한다.

**구현 난이도**: Medium
**예상 임팩트**: High
**영향 파일**:
- `components/InputSection.tsx`
- `app/page.tsx`
- `app/analyze/page.tsx`
- `components/MediaUploader.tsx`

**예상 작업량**: 3~4일

### Idea 2: 핸들러별 출력 스키마 계약 자동 검증 CI 통합 — 분석 품질 회귀 방지

**발견 근거**:
- `ef66147` 커밋에서 "핸들러 출력 스키마 검증"이 명시적으로 추가 — 스키마 불일치가 실운영에서 문제가 됐음을 시사
- `app/api/analyze/` 하위에 `figma.ts`, `url.ts`, `flow.ts`, `image.ts`, `comparison.ts` 5개 핸들러가 분리되어 있으나, 각 핸들러가 `outputSchema.ts`와 일치하는지 런타임에만 검증
- `app/api/analyze/route.ts`가 4회 변경으로 핫스팟 Top 3 — 핸들러 계약 변경이 라우터까지 반복적으로 파급
- QA 리포트: 타입 체크 `❌ FAIL` 지속 — 런타임 스키마 드리프트를 컴파일 타임에 잡지 못하는 구조

**현재 문제**:
- 핸들러가 새 필드를 추가하거나 기존 필드 타입을 변경해도, `outputSchema.ts`와의 불일치가 CI에서 즉시 감지되지 않음
- 스키마 검증 로직이 `route.ts`와 각 핸들러에 분산되어 중복 구현 및 누락 발생 가능
- 신규 핸들러(예: 향후 `voice.ts`, `pdf.ts` 추가 시) 계약 검증 기준이 명문화되지 않아 온보딩 비용 증가

**제안**:
핸들러 출력을 Zod 스키마로 단일 정의하고, CI 단계에서 **스키마 계약 테스트**를 자동 실행한다.

```typescript
// app/api/analyze/outputSchema.ts (확장)
import { z } from 'zod';

export const BaseOutputSchema = z.object({
  issues: z.array(IssueSchema),
  heatmap: HeatmapSchema,
  summary: z.string(),
  confidence: z.number().min(0).max(1),
});

// 핸들러별 확장 스키마
export const FigmaOutputSchema = BaseOutputSchema.extend({ ... });
export const UrlOutputSchema   = BaseOutputSchema.extend({ screenshotUrl: z.string().url() });
// ...

// app/api/analyze/handlers/base.ts (신규)
export function validateOutput<T extends z.ZodTypeAny>(
  schema: T,
  data: unknown,
  handlerName: string
): z.infer<T> {
  const result = schema.safeParse(data);
  if (!result.success) {
    // 구조화된 에러 로깅 + Sentry 전송
    throw new HandlerOutputError(handlerName, result.error);
  }
  return result.data;
}
```

```yaml
# .github/workflows/schema-contract.yml (신규)
- name: Handler Schema Contract Test
  run: npx ts-node scripts/validate-handler-contracts.ts
  # 각 핸들러 mock 입력 → 출력 스키마 자동 검증
```

**구현 난이도**: Medium
**예상 임팩트**: High
**영향 파일**:

- `app/api/analyze/route.ts`
- `app/api/analyze/handlers/figma.ts`
- `app/api/analyze/handlers/url.ts`
- `app/api/analyze/handlers/flow.ts`

### Idea 3: 분석 히스토리 세션 재개(Resume) 기능 — 중단된 분석 맥락 복원

**발견 근거**:
- `ef66147`에서 히스토리 타입 필터·재분석 기능이 추가 — 사용자가 과거 분석으로 돌아가 재작업하는 패턴이 실제로 존재함을 시사
- `4eacbca`에서 이슈 탭 화면별 필터링·썸네일 표시 개선 — 히스토리 탐색 UX에 반복 투자 중
- `app/history/page.tsx` + `app/report/[id]/page.tsx` + `app/share/[id]/page.tsx` 3개 페이지가 분리된 채 각자 히스토리 데이터를 소비 — 사용자 관점에서 흐름이 단절됨
- `067c53f`에서 localStorage 환경 전면 지원 — DB 없이도 히스토리가 쌓이므로, 로컬 세션 재개 수요가 실질적으로 존재

**현재 문제**:
- 사용자가 과거 분석 결과를 보다가 파라미터를 조정해 재분석하려면, 히스토리 → 메인 페이지로 수동 이동 후 설정을 처음부터 다시 입력해야 함
- 재분석(`ef66147`) 기능이 추가됐지만, 원본 입력(타깃 유저, 분석 목적, 업로드 파일 경로 등)이 히스토리에 저장되는지 여부가 불명확 — 재분석 시 컨텍스트 소실 위험
- 히스토리 아이템이 "결과 열람"용으로만 설계되어 있어, "이 분석에서 이어서 작업"하는 편집 플로우가 없음

**제안**:
히스토리 항목에 **입력 스냅샷**(분석 당시의 모든 파라미터)을 함께 저장하고, 리포트 페이지에서 "이 설정으로 재분석" 버튼 클릭 시 InputSection에 값을 복원하여 세션을 재개하는 Resume 플로우를 구현한다.

```typescript
// lib/history/sessionSnapshot.ts (신규)
export interface AnalysisSessionSnapshot {
  id: string;
  createdAt: string;
  inputMode: 'image' | 'url' | 'video' | 'figma' | 'comparison';
  params: {
    targetUser: string;
    analysisPurpose: string;
    deviceType: string;
    language: string;
  };
  inputRef: {
    type: 'file' | 'url';
    value: string;           // URL 또는 localStorage blob key
  };
  outputId: string;          // report/[id] 참조
}

// components/report/ResumeSessionButton.tsx (신규)
// 리포트 상단에 "이 설정으로 재분석" CTA 배치
// 클릭 시 sessionSnapshot을 app/page.tsx의 초기 상태로 주입
```

저장 위치: localStorage (`simulo:session:{id}`) — DB 미사용 환경과 호환 유지

**구현 난이도**: Low
**예상 임팩트**: Medium
**영향 파일**:
- `app/history/page.tsx`
- `app/report/[id]/page.tsx`
- `app/page.tsx`
- `components/ShareExportPanel.tsx`
