### Idea 1: API 응답 파싱 레이어 안정화 (Resilient Response Parser)

**발견 근거**: 
- 커밋 히스토리에서 반복되는 버그 패턴:
  - `c52252b` "fix: ensure all required fields exist after Claude response parse"
  - `7d47556` "fix: JSON parse failure when Claude appends trailing prose after closing brace"
  - `7dc4bed` "fix: comparison analysis JSON truncation and recovery"
  - `3ca6211` "fix: prevent localStorage quota exceeded on flow analysis"
- 최신 QA 리포트에서 "max_tokens 트런케이션"으로 인한 빌드 실패 언급
- `app/api/analyze/route.ts` (핫스팟 파일)의 파싱 로직이 엣지 케이스 처리 미흡

**현재 문제**: 
- Claude API 응답이 불완전하거나 JSON이 손상되면 전체 분석 파이프라인 실패
- localStorage 용량 초과로 인한 캐시 손실
- 각 API 엔드포인트마다 ad-hoc 파싱 로직으로 중복 코드 발생
- 사용자는 재시도 없이 "분석 실패" 메시지만 받음

**제안**: 
1. **전용 파서 모듈 생성** (`lib/claude/response-parser.ts`)
   - Claude 응답을 3단계로 처리: 텍스트 정규화 → JSON 추출 → 필드 검증
   - 손상된 JSON 복구 로직 (trailing 텍스트 제거, 누락된 괄호 자동 보완)
   - 필드별 fallback 값 정의

2. **재시도 메커니즘** 추가
   - API 호출 시 자동 재시도 (최대 3회)
   - 토큰 부족 시 프롬프트 자동 축소 후 재요청
   - 부분 성공 시 캐시하고 다음 호출 때 이어서 처리

3. **메모리 효율화**
   - 큰 응답을 청크 단위로 처리
   - IndexedDB 우선 사용 (localStorage 대체)
   - 일시적 데이터는 메모리에만 유지

**구현 난이도**: Medium

**예상 임팩트**: High
- fix 커밋 비율 20% 이상 감소 예상
- 분석 완료율 향상 (현재 추정: 85% → 95%)
- 개발자 디버깅 시간 50% 단축

**영향 파일**: 
- `lib/claude.ts` (리팩토링)
- `lib/claude/response-parser.ts` (신규)
- `app/api/analyze/route.ts`
- `app/api/analyze-flow/route.ts`
- `app/api/report/route.ts`

**예상 작업량**: 2-3일

### Idea 3: TypeScript 타입 안정성 강화 (CI/CD 게이트 자동화)

**발견 근거**: 
- 최신 QA 리포트 (2026-04-15):
  - 빌드: ⚠️ FAIL
  - 타입 체크: ❌ FAIL (TS2307: `@playwright/test` 모듈 미해석, TS7031: 암묵적 any)
- `e2e/settings.spec.ts`에서 `page` 매개변수 타입 미정의 (3곳)
- `tsconfig.json`과 `playwright.config.ts`의 설정 불일치
- `package-lock.json` 584줄 증가로 의존성 버전 충돌 위험

**현재 문제**: 
- CI 파이프라인 실패로 인한 배포 지연
- 로컬 개발 환경 ≠ CI 환경 → "로컬에선 되는데 CI에선 안 됨" 문제 반복
- IDE 자동완성 미지원 → 개발 생산성 저하
- 런타임 에러 가능성 미리 감지 불가능
- E2E 테스트 자체의 타입 신뢰도 저하

**제안**: 
1. **TypeScript 설정 통일**
   ```json
   // tsconfig.json
   {
     "compilerOptions": {
       "skipLibCheck": false,
       "moduleResolution": "node",
       "resolveJsonModule": true,
       "types": ["@playwright/test", "node"],
       "strict": true
     },
     "include": ["app", "lib", "components", "e2e"]
   }
   ```
   - Playwright 타입 명시적 포함
   - strict 모드 활성화

2. **공통 타입 정의 모듈 생성** (`types/index.ts`)
   ```typescript
   // types/index.ts
   export type { Page, BrowserContext, Test } from '@playwright/test';
   export interface AnalysisResult { /* ... */ }
   export interface ReportData { /* ... */ }
   ```
   - 모든 API/컴포넌트가 공통 타입 참조

3. **CI 게이트 자동화**
   - `npm run type-check` 실패 시 PR 블로킹
   - `npm run type-check:e2e` 별도 검증
   - 린트 + 타입 체크를 `npm run validate`로 통합
   - GitHub Actions 워크플로우에 타입 체크 단계 추가 (이미 존재하지만 실패 무시 중)

4. **E2E 테스트 타입 표준화**
   ```typescript
   // e2e/fixtures.ts (신규)
   import { test as base, Page } from '@playwright/test';
   
   export const test = base.extend({
     // 타입 추론이 자동으로 작동
   });
   ```
   - 모든 E2E 테스트가 `e2e/fixtures.ts`의 `test` 사용

**구현 난이도**: Low

**예상 임팩트**: Medium
- CI 실패율 0% (현재 추정: 10~15%)
- 개발자 IDE 경험 개선 → 코드 작성 속도 15% 향상
- 런타임 에러 사전 차단 → 핫픽스 빈도 30% 감소
- QA 리포트의 타입 체크 status "❌ FAIL" → "✅ PASS"

**영향 파일**: 
- `tsconfig.json`
- `tsconfig.e2e.json` (신규)
- `types/index.ts` (신규)
- `e2e/fixtures.ts` (신규)
- `e2e/api.spec.ts`
- `e2e/pages.spec.ts`
- `e2e/settings.spec.ts`
- `.github/workflows/auto-qa-ideas.yml`

**예상 작업량**: 1-2일
