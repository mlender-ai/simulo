### Idea 1: analyze route.ts 오케스트레이터 경량화 — 핸들러 자동 디스패치 레지스트리

**발견 근거**:
- `app/api/analyze/route.ts`가 핫스팟 1위(6회 변경). 동시에 `app/api/analyze/registry.ts`가 이미 존재하고 핸들러 4종(figma/comparison/flow/image)이 분리됐음에도 route.ts가 계속 수정되는 패턴 → 오케스트레이터가 핸들러 선택·에러 위임·응답 직렬화를 여전히 직접 처리하고 있다는 신호
- `feat: analyze 핸들러 분리`(e6f51fd) 이후에도 route.ts 수정이 3회 추가로 발생 — 분리 효과가 희석되고 있음
- `app/api/analyze/handlers/types.ts` 존재 → 타입은 공유되나 디스패치 계약(contract)이 느슨함

**현재 문제**:
- `registry.ts`가 등록만 하고 실제 라우팅 결정(어떤 핸들러를 언제 호출할지)은 `route.ts`에 인라인으로 남아있을 가능성 — 새 핸들러(예: 비디오) 추가 시 route.ts를 필수 수정해야 하는 구조
- 핸들러별 에러 포맷·응답 스키마가 통일되지 않으면 route.ts가 각 케이스별 분기 처리를 흡수하게 됨

**제안**:
```typescript
// app/api/analyze/registry.ts (확장)
type HandlerDescriptor = {
  match: (payload: AnalyzePayload) => boolean; // 조건 선언
  handle: (payload: AnalyzePayload, ctx: HandlerContext) => Promise<AnalyzeResult>;
  name: string;
};

// route.ts는 단 하나의 역할만 수행
const handler = registry.find(h => h.match(payload));
if (!handler) return standardError('NO_HANDLER');
return await dispatcher.run(handler, payload); // 에러·응답 직렬화는 dispatcher 담당
```
- `registry.ts`에 `match()` 조건을 선언적으로 기술 → route.ts는 dispatcher 호출 1줄로 축소
- `dispatcher.ts` 신설: 핸들러 실행, try/catch, 응답 직렬화(표준 envelope), 실행 시간 로깅 통합
- 새 입력 타입(예: 비디오, 코드) 추가 시 registry에만 항목 추가 → route.ts 무변경

**구현 난이도**: Medium
**예상 임팩트**: High
**영향 파일**:
```

app/api/analyze/registry.ts
app/api/analyze/handlers/types.ts
app/api/analyze/handlers/figma.ts
app/api/analyze/handlers/comparison.ts
app/api/analyze/handlers/flow.ts
app/api/analyze/handlers/image.ts
app/api/analyze/dispatcher.ts (신규)
```
**예상 작업량**: 2~3일

### Idea 2: 멀티모달 입력 진입점 분기 UX — "무엇을 분석할지" 먼저 선택하는 Intent-First 플로우

**발견 근거**:
- 일주일 내 **영상 업로드**(39781f4), **코드→Figma 역방향 분석**(18f1802), **Action Export Hub**(64b7a63)가 동시 출시 → 지원 입력 타입이 이미지·영상·코드·Figma URL로 4종이 됨

- `fix: PNG 업로드 에러 수정`(7b9cded) — 입력 타입 증가에 따른 업로드 경로별 엣지케이스 버그 발생 중
- `app/page.tsx`(5회 변경)가 핫스팟 3위 — 랜딩/메인 입력 화면이 기능 추가마다 수정되는 불안정한 진입점

- 입력 타입별로 요구되는 부가 파라미터(타깃 유저, 분석 깊이 등)가 다름에도 동일한 InputSection이 조건부 렌더링으로 처리 → 조건 분기가 824줄 컴포넌트를 더 팽창시키는 악순환

```

│  무엇을 분석하시겠어요?                   │
│                                         │
│  [ 🖼  화면 이미지 ]  [ 🎬 사용 영상 ]   │
│  [ </> 코드 리뷰  ]  [ 🔗 Figma URL  ]  │
```

- 선택된 타입에 따라 **전용 InputSection 서브컴포넌트**(`ImageInput`, `VideoInput`, `CodeInput`, `FigmaInput`)로 라우팅 → 각 컴포넌트는 해당 타입에만 필요한 필드만 렌더링
- `app/page.tsx`의 분기 로직 제거 → `app/analyze/[type]/page.tsx` 구조로 URL 기반 타입 명시 (히스토리·공유 링크에서도 분석 타입 복원 가능)

### Idea 3: E2E 테스트 부트스트랩 자동화 — CI 타입 오류 자가 복구 가드레일

**발견 근거**:
- QA 리포트 기준 **TypeScript 타입 체크 FAIL + 빌드 FAIL**이 이번 리포트에서 확인되며, 동일 문제(`@playwright/test` 미등록, 암묵적 `any`)가 **2주 이상 지속** 미해결 상태
- 이전 아이디어(20260415, 20260416)에서 "대용량 파일 분리"와 "파싱 레이어 표준화"를 다뤘으나, **E2E 테스트 인프라 자체의 셋업 자동화**는 미다룬 영역
- `a11d3cb` 커밋(ci: auto-qa-ideas 워크플로우 수정)이 있음에도 QA 자동화가 Playwright 타입 오류를 사전 차단하지 못함 → CI가 증상을 리포트하지만 원인을 예방하지 못하는 구조
- `770c8c0 fix: add exponential backoff retry to all Claude API calls` — API 레이어엔 재시도 로직이 있으나, 테스트 인프라엔 동일 수준의 견고성 부재

**현재 문제**:
- `@playwright/test` 타입 선언 누락으로 E2E 3개 파일 전체가 타입 에러 — 테스트가 런타임에선 통과해도 타입 체크 단계에서 CI 전체를 블로킹
- `tsconfig.json`과 `playwright.config.ts`의 모듈 해석 설정이 비동기화 → 로컬 통과 / CI 실패 패턴 반복 가능성
- `e2e/fixtures.ts`가 부재 — 각 테스트 파일이 독립적으로 `page` 타입을 선언해야 하므로 동일한 암묵적 `any` 패턴이 새 테스트마다 재발

**제안**:
1. **`e2e/fixtures.ts` 표준 베이스 픽스처 신설**:
```typescript
// e2e/fixtures.ts
import { test as base, Page } from '@playwright/test';

export type SimuloFixtures = {
  analyzePage: Page;       // /analyze 진입 완료 상태
  withHistory: Page;       // 히스토리 1건 이상 존재 상태
};

export const test = base.extend<SimuloFixtures>({
  analyzePage: async ({ page }, use) => {
    await page.goto('/analyze');
    await use(page);
  },
});
export { expect } from '@playwright/test';
```

2. **`tsconfig.playwright.json` 분리** (빌드 tsconfig와 충돌 방지):
```json
{
  "extends": "./tsconfig.json",
  "compilerOptions": {
    "types": ["@playwright/test", "node"],
    "skipLibCheck": false
  },
  "include": ["e2e/**/*", "playwright.config.ts"]
}
```

3. **CI pre-check 스크립트 추가** (`package.json`):
```json
"scripts": {
  "test:type-e2e": "tsc --project tsconfig.playwright.json --noEmit",
  "pretest:e2e": "npm run test:type-e2e"  // E2E 실행 전 타입 검사 강제
}
```
- GitHub Actions 워크플로우에 `test:type-e2e` 스텝 추가 → 타입 오류 시 E2E 실행 전 조기 실패·명확한 에러 메시지 제공
- `@playwright/test` 버전을 `package.json` `devDependencies`에 고정(pinning) + Renovate/Dependabot 설정으로 자동 업데이트

**구현 난이도**: Low
**예상 임팩트**: Medium
**영향 파일**:
```

e2e/settings.spec.ts
e2e/api.spec.ts
e2e/pages.spec.ts
tsconfig.playwright.json (신규)

```
**예상 작업량**: 0.5~1일
