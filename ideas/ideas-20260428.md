### Idea 1: 히트맵 존 피커 검증 루프 — 사용자 피드백 없는 단방향 변경의 위험 제거

**발견 근거**:
- 히트맵 존 정확도 관련 fix 커밋 2건(e72e3cc, fe23647) 이후 단일 존 피커 아키텍처로 전환(f353f91)이 이루어졌으나, 이 전환이 "기술적 수렴"인지 "UX 후퇴"인지 검증할 수단이 현재 없음
- `HeatmapViewer.tsx`가 4회 수정 — 해당 컴포넌트가 렌더링과 인터랙션 로직을 모두 담고 있어 변경마다 부작용 범위가 넓음

**현재 문제**:
- 기존 다중 존 선택 → 단일 존 피커로 전환 시, 복수의 UX 이슈 영역을 동시에 마킹하던 사용자는 워크플로우가 단절됨
- `HeatmapViewer.tsx` 내부에 렌더링 로직(그리드 기반 좌표 계산), 상태 관리(선택된 존), UI(피커 오버레이)가 혼재하여 추가 정확도 버그 발생 시 격리가 어려움
- 단일 존 피커의 정확도 개선이 실제 분석 품질로 이어지는지 측정 지표 없음

**제안**:
1. **`HeatmapViewer.tsx` 관심사 분리**: 좌표 계산 로직을 `lib/heatmap/zoneCalculator.ts`로 추출하고, 피커 오버레이를 `components/heatmap/ZonePicker.tsx`로 분리하여 HeatmapViewer는 조합만 담당
2. **존 선택 후 인라인 확인 단계 추가**: 사용자가 존을 선택하면 "이 영역이 맞나요? [확인 / 다시 선택]" 미니 confirm UI를 노출하여 오선택 방지 및 사용자 의도 데이터 수집
3. **다중 존 순차 선택 모드 플래그 추가**: 단일 존 피커를 기본으로 유지하되, "여러 영역 분석" 토글로 순차 존 추가(최대 3개) 기능을 선택적으로 활성화 — 단방향 아키텍처 전환의 UX 손실을 보완

```
components/
└── heatmap/
    ├── HeatmapViewer.tsx     (조합 컨테이너, 현재 대비 -60% 코드)
    ├── ZonePicker.tsx        (신규 — 피커 오버레이 UI)
    └── ZoneConfirmTooltip.tsx (신규 — 인라인 확인 UI)
lib/heatmap/
└── zoneCalculator.ts         (신규 — 그리드 좌표 계산 순수 함수)
```

**구현 난이도**: Medium
**예상 임팩트**: High
**영향 파일**:
- `components/HeatmapViewer.tsx` (분리 대상)
- `lib/heatmap/zoneCalculator.ts` (신규)
- `components/heatmap/ZonePicker.tsx` (신규)
- `components/heatmap/ZoneConfirmTooltip.tsx` (신규)
**예상 작업량**: 2~3일

### Idea 2: 히스토리 페이지 데이터 레이어 분리 — 핫스팟 파일의 파편화 수렴

**발견 근거**:
- `app/history/page.tsx`가 7일간 4회 수정 — 날짜 그룹핑(b9b8fb9), 타입 필터 + 재분석 + 핸들러 검증(ef66147), PNG 버튼 제거(8387945), 썸네일 개선(4eacbca)이 모두 단일 페이지 파일에 직접 커밋됨
- 현재 히스토리 페이지는 **데이터 페칭 + 필터링 상태 + 날짜 그룹핑 로직 + 썸네일 렌더링 + 재분석 트리거 + 타입 필터 UI**가 하나의 파일에 공존 — 이전 아이디어와 달리 "기능 추가"가 아닌 "로직 분리"에 초점

**현재 문제**:
- 날짜 그룹핑 로직(오늘/어제/이번 주/이전 분류)이 페이지 컴포넌트 내부에 인라인으로 작성되어 있어 히스토리 진입 시마다 재계산됨 — 별도 유틸 또는 `useMemo` 없이 렌더 함수 내 직접 실행될 가능성 높음
- 타입 필터(이미지/URL/영상/피그마 등)와 날짜 필터가 각각 독립 state로 관리되면서, 복합 필터 상태를 `app/api/history/route.ts`가 아닌 클라이언트에서 처리 — 대량 히스토리 시 성능 저하 위험
- 재분석 트리거가 히스토리 페이지에 포함되어 있어, 히스토리 조회 + 분석 실행의 두 가지 책임을 동시에 가짐

**제안**:
1. **`useHistory` 커스텀 훅 신설**: 필터 상태(타입 + 날짜 범위), 날짜 그룹핑 계산, API 페칭을 훅으로 추출 — 페이지 컴포넌트는 UI 렌더링만 담당
2. **날짜 그룹핑을 서버사이드로 이동**: `app/api/history/route.ts`가 `group: "today" | "yesterday" | "this_week" | "older"` 필드를 포함한 응답을 반환하도록 변경 — 클라이언트 계산 제거
3. **`HistoryCard.tsx` 컴포넌트 분리**: 썸네일 + 타입 배지 + 재분석 버튼을 카드 단위로 캡슐화하여 히스토리 페이지는 리스트 컨테이너 역할만 수행

```
app/history/
└── page.tsx                  (UI 컨테이너, -70% 코드)
hooks/
└── useHistory.ts             (신규 — 필터·페칭·그룹핑)
components/history/
└── HistoryCard.tsx           (신규 — 카드 단위 컴포넌트)
app/api/history/
└── route.ts                  (그룹핑 로직 흡수)
```

**구현 난이도**: Low
**예상 임팩트**: Medium
**영향 파일**:
- `app/history/page.tsx` (핵심 리팩토링 대상)
- `app/api/history/route.ts` (그룹핑 로직 이전)
- `hooks/useHistory.ts` (신규)
- `components/history/HistoryCard.tsx` (신규)
**예상 작업량**: 1~2일

### Idea 3: 대시보드 데이터 계약 레이어 — localStorage/DB 이중 환경의 구조적 안정화

**발견 근거**:
- 인사이트 대시보드(5193ae1) 출시 후 **2개의 fix 커밋이 즉각 후속**(056fe08 localStorage fallback, 7b7579e 무한 스피너 + null 안전성) — 데이터 소스 환경(DB vs localStorage)에 따른 방어 코드가 사후 패치로 누적되는 패턴
- `app/dashboard/page.tsx`가 3회 수정으로 핫스팟 공동 4위 — 대시보드는 `app/api/dashboard/stats/route.ts`와 `app/api/dashboard/insights/route.ts` 두 API에 의존하며, 두 API 모두 localStorage 환경에서의 동작이 명세화되지 않은 채 운영 중
- 이전 아이디어(26일자 리포트)에서 localStorage 전환 후 멀티-입력 타입 간 리포트 일관성 미검증 문제가 지적되었으나, 대시보드 계층의 데이터 계약 자체가 불명확한 근본 원인은 미해결

**현재 문제**:
- `app/api/dashboard/stats/route.ts`와 `insights/route.ts`가 DB 접근 시도 → 실패 → localStorage fallback의 흐름을 각자 독립적으로 구현할 가능성이 높아, fallback 로직의 일관성이 API 파일마다 다름
- 대시보드 페이지에서 두 API를 병렬 호출하는 경우, 한쪽 API만 null을 반환해도 전체 대시보드가 무한 스피너에 빠질 수 있는 구조 — 7b7579e fix의 반복 가능성 잠재
- AI 역제안 기능이 대시보드에 포함되어 있으나, localStorage 환경에서 충분한 데이터 없이 역제안을 생성할 때의 폴백 메시지나 최소 데이터 요건이 정의되지 않음

**제안**:
1. **데이터 소스 어댑터 패턴 도입**: `lib/dashboard/dataSource.ts`에 `DashboardDataSource` 인터페이스를 정의하고, `LocalStorageDataSource`와 `DatabaseDataSource`를 구현체로 분리 — API 라우트는 환경에 따라 적절한 구현체를 주입받아 사용

```typescript
// lib/dashboard/dataSource.ts
export interface DashboardDataSource {
  getStats(): Promise<DashboardStats>;
  getInsights(): Promise<DashboardInsight[]>;
}

export class LocalStorageDataSource implements DashboardDataSource { ... }
export class DatabaseDataSource implements DashboardDataSource { ... }

// 환경 판별 팩토리
export function createDataSource(): DashboardDataSource {
  return process.env.DATABASE_URL 
    ? new DatabaseDataSource() 
    : new LocalStorageDataSource();
}
```

2. **`DashboardStats` / `DashboardInsight` 타입 계약 명세화**: `app/api/dashboard/outputSchema.ts` 신설 (analyze의 `outputSchema.ts` 패턴 동일 적용) — null 허용 필드와 필수 필드를 Zod 스키마로 명시하여 null 안전성 문제를 런타임이 아닌 스키마 레벨에서 차단

3. **AI 역제안 최소 데이터 가드 추가**: 분석 히스토리가 N개(예: 3개) 미만인 경우 역제안 API 호출 대신 "분석을 더 진행하면 인사이트를 제공합니다" 상태를 반환하는 조건부 로직 추가 — 빈 데이터 기반 역제안으로 인한 품질 저하 방지

```
lib/dashboard/
├── dataSource.ts             (신규 — 어댑터 인터페이스 + 팩토리)
├── LocalStorageDataSource.ts (신규)
└── DatabaseDataSource.ts     (신규)
app/api/dashboard/
├── outputSchema.ts           (신규 — Zod 스키마)
├── stats/route.ts            (dataSource 주입 방식으로 변경)
└── insights/route.ts         (dataSource 주입 방식으로 변경)
app/dashboard/
└── page.tsx                  (null 처리 단순화)
```

**구현 난이도**: Medium
**예상 임팩트**: High
**영향 파일**:
- `app/dashboard/page.tsx`
- `app/api/dashboard/stats/route.ts`
- `
