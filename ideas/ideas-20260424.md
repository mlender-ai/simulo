### Idea 1: 입력 검증 로직의 선언적 스키마 분리 — 에러 UI 재발 방지

**발견 근거**:
- 7일 내 필수 필드 에러 관련 fix 커밋 2건 연속 (90d6357 "에러 박스 + 필드 강조", 7b9cded "필수 필드 에러 표시") — 동일 레이어에서 버그가 반복되는 패턴
- `components/InputSection.tsx`가 핫스팟 5위(변경 3회)이며 824줄 단일 파일로 검증 로직·UI·상태관리가 혼재
- 타깃 유저 기본값 수정(7b9cded)이 같은 커밋에 묶인 것으로 보아, 검증 로직이 컴포넌트 내부에 하드코딩된 구조로 추정

**현재 문제**:
- 필수 필드 목록, 기본값, 에러 메시지가 컴포넌트 코드 내부에 분산되어 있어 한 곳 수정 시 다른 케이스가 누락됨
- 새 입력 필드(영상, Figma URL 등) 추가 시마다 검증 로직을 수동으로 추가해야 하는 확장성 부재
- 에러 상태(border 강조 / 에러 박스 / 버튼 비활성화)가 여러 위치에서 개별 관리되어 일관성 깨짐

**제안**:
1. `lib/validation/inputSchema.ts` 신설 — Zod 스키마로 필드 정의, 필수 여부, 기본값, 에러 메시지를 단일 소스로 선언
2. `hooks/useInputValidation.ts` 추출 — 컴포넌트에서 검증 로직 완전 분리, `validate()` 호출 시 필드별 에러 맵 반환
3. `components/input/FieldErrorHighlight.tsx` 공통 컴포넌트화 — 에러 border·메시지 표시를 단일 진실 공급원에서 렌더링
4. 신규 입력 타입(비디오, Figma, 코드) 추가 시 스키마만 확장하면 에러 UI 자동 반영되는 구조 확보

```
lib/validation/
├── inputSchema.ts       ← Zod 스키마 + 기본값 선언
└── inputSchema.test.ts  ← 단위 테스트

hooks/
└── useInputValidation.ts

components/input/
├── InputSection.tsx     ← 컨테이너 (검증 로직 제거)
└── FieldErrorHighlight.tsx
```

**구현 난이도**: Medium
**예상 임팩트**: High
**영향 파일**:
- `components/InputSection.tsx`
- `app/page.tsx`
- `app/analyze/page.tsx`
- (신규) `lib/validation/inputSchema.ts`
- (신규) `hooks/useInputValidation.ts`

### Idea 2: 멀티 미디어 입력 파이프라인 통합 — MediaUploader 확장성 리팩토링

**발견 근거**:
- 7일 내 영상 업로드(`app/api/upload-video/route.ts`, 39781f4), PNG 업로드 에러 수정(7b9cded), OCR 파이프라인 신설(2400afd), 코드→Figma 역방향 분석(18f1802)이 개별적으로 추가됨
- `components/MediaUploader.tsx`가 핫스팟 6위(변경 3회)로, 미디어 타입이 추가될 때마다 파일이 직접 수정되는 패턴
- `app/api/analyze/` 하위에 `handlers/figma.ts`, `handlers/image.ts`, `handlers/flow.ts` 등 핸들러가 분산되어 있으나 라우팅 로직은 여전히 `route.ts`(핫스팟 2위, 변경 6회)에 집중

**현재 문제**:
- 이미지 / 비디오 / OCR / Figma URL / 코드 입력이 각기 다른 API 엔드포인트로 분산되어 있어 업로드 전 검증(Pre-flight), 진행 상태 표시, 에러 처리가 미디어 타입마다 중복 구현됨
- `MediaUploader.tsx`가 지원 타입 증가에 따라 조건 분기가 누적 → 새 미디어 타입 추가 시 컴포넌트·API 양쪽 모두 수정 필요
- 비디오 프레임 분석 결과와 이미지 OCR 결과가 별도 상태로 관리되어 최종 리포트 생성 시 병합 로직 복잡도 증가 우려

**제안**:
1. `lib/media/mediaTypeRegistry.ts` 신설 — 지원 미디어 타입(image/video/figma/code)을 등록 방식으로 선언, 각 타입별 허용 확장자·최대 크기·처리 엔드포인트를 중앙 관리
2. `app/api/analyze/route.ts`의 미디어 타입 분기를 registry 기반 동적 핸들러 호출로 전환 (현재 `registry.ts` 존재 확인 — 기존 구조 활용)
3. `components/MediaUploader.tsx`를 registry를 읽어 UI를 동적 렌더링하는 구조로 전환 — 새 타입은 registry 등록만으로 업로더 UI 자동 확장
4. 업로드 진행 상태·에러를 `useMediaUpload` 훅으로 통합 관리, 타입별 분기 없이 공통 상태 인터페이스 사용

```
lib/media/
├── mediaTypeRegistry.ts     ← 타입별 설정 등록
├── useMediaUpload.ts        ← 공통 업로드 훅
└── processors/
    ├── imageProcessor.ts
    ├── videoProcessor.ts
    └── figmaProcessor.ts

app/api/analyze/
├── route.ts                 ← registry 기반 라우팅으로 경량화
└── handlers/                ← 기존 핸들러 유지, 인터페이스 통일
```

**구현 난이도**: High
**예상 임팩트**: High
**영향 파일**:
- `app/api/analyze/route.ts`
- `app/api/analyze/registry.ts`
- `components/MediaUploader.tsx`
- `app/api/upload-video/route.ts`
- `app/api/ocr-extract/route.ts`
- (신규) `lib/media/mediaTypeRegistry.ts`

**예상 작업량**: 4~5일

### Idea 3: Export 파이프라인 상태 관리 UI — Action Export Hub 사용성 완성

**발견 근거**:
- 64b7a63 커밋에서 `Action Export Hub + Plugin Registry + Pre-flight Validation`이 한 번에 추가됨 — 기능은 구현됐으나 대규모 단일 커밋은 UX 마찰이 검증되지 않은 채 릴리즈된 신호
- `app/api/export/` 하위에 docx / md / jira / pdf 4개 엔드포인트가 존재하나, `components/ShareExportPanel.tsx` 단일 컴포넌트가 모든 export 타입을 처리하는 구조
- QA 리포트 기준 빌드 FAIL 상태에서 Export 기능의 E2E 테스트가 존재하지 않아 실제 export 성공 여부가 검증되지 않음

**현재 문제**:
- 4개 export 포맷(DOCX, MD, JIRA, PDF)의 생성 진행 상태·성공·실패가 사용자에게 명확히 피드백되지 않을 가능성 (Pre-flight Validation 추가됐으나 export 진행 중 상태 표시는 미확인)
- JIRA export는 외부 API 연동으로 네트워크 지연·인증 실패 시 사용자 대기 경험이 다른 포맷(로컬 생성)과 동일 UI로 처리될 경우 혼란 발생
- Export 성공 후 다운로드 파일명, 타임스탬프, 포맷별 포함 섹션이 사용자에게 노출되지 않아 "무엇이 내보내졌는가"에 대한 확인 수단 부재

**제안**:
1. **Export 상태 피드백 강화**: 각 export 버튼에 `idle → validating → generating → ready → error` 5단계 상태를 시각화하는 `useExportStatus` 훅 추가, JIRA는 별도 "연결 중" 단계 표시
2. **Export Preview 패널**: 다운로드 전 "이 리포트에 포함될 섹션" 체크리스트를 모달로 표시 — 사용자가 포함/제외 섹션을 선택한 후 export 실행 (DOCX·PDF 대상 우선)
3. **Export 이력 로깅**: `app/history/page.tsx`에 분석 이력과 함께 export 이력(포맷, 생성 시각, 성공 여부)을 표시 — `app/api/history/route.ts` 확장으로 구현
4. **E2E 테스트 추가**: `e2e/export.spec.ts` 신설, 4개 포맷별 export 성공 케이스 및 Pre-flight Validation 실패 케이스 커버

```
hooks/
└── useExportStatus.ts

components/
├── ShareExportPanel.tsx     ← 상태 피드백 UI 추가
└── export/
    ├── ExportPreviewModal.tsx
    └── ExportHistoryItem.tsx

e2e/
└── export.spec.ts           ← 신규 테스트

app/api/history/route.ts     ← export 이력 필드 확장
```

**구현 난이도**: Medium
**예상 임팩트**: Medium
**영향 파일**:
- `components/ShareExportPanel.tsx`
- `app/api/export/docx/[id]/route.ts`
- `app/api/export/pdf/[id]/route.ts`
- `app/api/export/jira/[id]/route.ts`
- `app/api/export/md/[id]/route.ts`
- `app/api/history/route.ts`
- `app/history/page.tsx`
- (신규) `hooks/useExportStatus.ts`
- (신규) `e2e/export.spec.ts`

**예상 작업량**: 3~4일
