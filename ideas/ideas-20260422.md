## Idea 1: TypeScript 빌드 안정성 개선 — 모듈 해석 및 타입 정의 자동화

**발견 근거**: 
- QA 리포트에서 **타입 체크 FAIL** (TS2307, TS7031)
- `@playwright/test` 모듈 미인식, E2E 테스트 파일 3개(`api.spec.ts`, `pages.spec.ts`, `settings.spec.ts`)에서 암묵적 `any` 타입
- `playwright.config.ts`에서 모듈 해석 실패로 build 파이프라인 차단

**현재 문제**: 
- 로컬 개발과 CI/CD 환경 간 타입 검사 불일치 (tsconfig.json과 tsconfig.e2e.json 동기화 부재)
- E2E 테스트 매개변수(`page`, `context`)에 명시적 타입 선언 없어 자동완성 및 타입 안정성 저하
- 빌드 시점에 런타임 에러가 아닌 타입 에러로 배포 지연

**제안**: 
1. **통합 타입 정의 파일 생성** (`types/test.d.ts`):
   ```typescript
   import { Page, BrowserContext, Test } from '@playwright/test';
   declare global {
     namespace PlaywrightTest {
       interface TestFixtures {
         authenticatedPage: Page;
       }
     }
   }
   ```
2. **tsconfig.json 업데이트**:
   - `skipLibCheck: true` 추가
   - `moduleResolution: "node"` 명시
   - `types: ["@playwright/test", "node"]` 포함
3. **E2E 테스트 기본 설정 표준화** (`e2e/fixtures.ts` 신규 작성):
   - 모든 E2E 테스트가 공통 fixtures 상속하도록 리팩토링
   - 자동 타입 추론 및 IDE 지원 활성화

**구현 난이도**: Medium
**예상 임팩트**: High
**영향 파일**: 
- `tsconfig.json`
- `tsconfig.e2e.json` (신규 또는 통합)
- `playwright.config.ts`
- `e2e/fixtures.ts` (신규)
- `types/test.d.ts` (신규)
- `e2e/*.spec.ts` (3개 파일 마이그레이션)

**예상 작업량**: 4-5시간 (타입 정의 작성 + E2E 테스트 수정 + 검증)

## Idea 2: 대용량 핫스팟 파일 단계별 모듈화 — 유지보수성 개선

**발견 근거**: 
- `lib/claude.ts` 893줄, `components/InputSection.tsx` 824줄, `components/ReportTabs.tsx` 811줄
- 이 3개 파일이 Git log에서 가장 많이 변경됨 (각 4회 이상 수정)
- QA 리포트에서 "단일 책임 원칙(SRP) 위반"으로 테스트 작성 어려움 지적

**현재 문제**: 
- 코드 변경 시 부작용 위험 증가 (특정 기능 수정 시 전체 컴포넌트 영향)
- 테스트 작성 및 모킹이 복잡함 (단일 파일에 여러 책임 혼재)
- 협업 시 merge conflict 빈도 증가
- 최근 7일간 `3-Pass OCR 보정 파이프라인`, `비디오 업로드`, `코드→Figma 역방향 분석` 기능 추가로 인해 핫스팟 파일이 더욱 비대화됨

**제안**: 
1. **`lib/claude.ts` 분리** (단계 1):

   ```
   ├── ai-client.ts       (Claude API 호출, retry 로직)
   ```

2. **`components/InputSection.tsx` 분해** (단계 2):

   ```
   components/input/
   ├── InputSection.tsx       (컨테이너)
   ├── FileUploader.tsx       (파일 업로드 UI)
   ├── ParameterControl.tsx   (OCR/분석 파라미터)
   ├── PresetSelector.tsx     (프리셋 선택)
   └── hooks/useInputState.ts (상태 관리)
   ```
3. **`components/ReportTabs.tsx` 탭별 분리** (단계 3):

   ```
   ├── ReportTabs.tsx     (탭 라우터)

   │   ├── OverviewTab.tsx
   │   ├── DetailTab.tsx
   │   └── ComparisonTab.tsx
   ```

**구현 난이도**: High
**예상 임팩트**: High

## Idea 3: 반복 에러 방지 시스템 구축 — 에러 패턴 추적 및 자동 검증

**발견 근거**: 
- `.claude/memory/errors.md`, `.claude/memory/patterns.md` 파일 미정의
- 최근 7일 커밋 중 **fix 커밋 비율이 7%**로 매우 낮음 (총 29개 중 2개)
- QA 리포트에서 "TS2307, TS7031" 에러가 반복되었을 가능성이 높으나 추적 기록 없음
- `770c8c0 fix: add exponential backoff retry to all Claude API calls` 커밋 이후로도 유사한 API 통신 에러 패턴이 반복될 수 있음

**현재 문제**: 
- 동일한 빌드 에러(타입, 모듈), API 에러(timeout, rate limit)가 재발할 수 있음
- 개발자 간 에러 정보 공유 부재
- QA 에이전트가 발견한 이슈(e2e 테스트 타입 안정성, package-lock 복잡성)가 다음 스프린트에서 반복될 위험
- 자동화된 검증이 부족하여 같은 이슈로 여러 번 빌드 실패 가능

**제안**: 
1. **에러 추적 시스템** (`.claude/memory/errors.md`):
   ```markdown
   # Error Registry
   
   ## Pattern: TS2307 (Module not found)
   - 원인: @playwright/test 타입 정의 누락
   - 해결: npm install --save-dev @playwright/test@latest
   - 예방: postinstall 스크립트에 타입 검증 추가
   - 발생일: 2026-04-15, 2026-04-20
   
   ## Pattern: API Timeout (Claude)
   - 원인: 대용량 이미지 분석 시 30초 초과
   - 해결: exponential backoff retry + 타임아웃 60초로 확대
   - 예방: lib/claude/ai-client.ts에서 요청 크기 사전 검증
   ```

2. **자동 검증 스크립트** (`scripts/validate-patterns.ts` 신규):
   ```typescript
   // 빌드 전 자동 실행
   - TypeScript 모듈 해석 검증
   - package.json 의존성 검증
   - E2E 테스트 타입 정의 검증
   - 대용량 파일(>800줄) 감지 알림
   ```

3. **QA 에이전트 개선** (`scripts/agents/qa-agent.ts`):
   - 이전 에러 기록과 비교 (에러 재발 감지)
   - 패턴 매칭으로 동일 이슈 자동 분류
   - 자동 아이디어 리포트에 "Recurring Issues" 섹션 추가

**구현 난이도**: Low
**예상 임팩트**: Medium
**영향 파일**: 
- `.claude/memory/errors.md` (신규)
- `.claude/memory/patterns.md` (신규)
- `scripts/validate-patterns.ts` (신규)
- `scripts/agents/qa-agent.ts` (수정)
- `package.json` (postinstall 또는 validate 스크립트 추가)

**예상 작업량**: 6-8시간 (에러 카탈로그 작성 3h + 자동화 스크립트 3-5h)
