## Idea 1: Claude API 응답 파싱 안정성 레이어 추상화

**발견 근거**: 
- Git log에서 반복되는 JSON 파싱 에러 수정: `fix: JSON parse failure when Claude appends trailing prose after closing brace`, `fix: ensure all required fields exist after Claude response parse`, `fix: prevent localStorage quota exceeded on flow analysis`
- 최신 QA 리포트에서 "flow analysis JSON parse failures from truncation" 지적
- `response-parser.ts`가 핫스팟(3회 변경) → 파싱 로직이 자주 깨짐
- fix 커밋 20개 중 ~6개가 Claude 응답 처리 관련

**현재 문제**: 
- Claude API 응답이 토큰 제한으로 잘려도 JSON 유효성 검증 없음
- 필드 누락/undefined 가드가 산발적으로 분산 (ReportTabs, ComparisonReportTabs 등)
- 같은 에러가 여러 곳에서 반복되어 수정 비용 증가 (중복 코드)
- localStorage quota 초과로 인한 조용한 실패 가능성

**제안**:
```typescript
// lib/claude/response-validator.ts (신규)
export class SafeResponseParser {
  static parse<T>(raw: string, schema: z.ZodSchema<T>): T | null {
    // 1. 잘린 JSON 자동 복구 (trailing bracket 추가)
    // 2. Zod 스키마로 자동 검증 및 기본값 주입
    // 3. 실패 시 fallback 값 반환
    // 4. 파싱 에러 로깅 (모니터링용)
  }
}

// 모든 API 라우트에서 통일된 사용
export async function POST(req: Request) {
  const raw = await claude.messages.create(...)
  const validated = SafeResponseParser.parse(raw.content, AnalysisSchema)
  if (!validated) return { issues: [], thinkAloud: "" } // 안전한 기본값
}
```

**구현 난이도**: Medium

**예상 임팩트**: High
- fix 커밋 비율을 현재 41% → 15% 이하로 감소
- 런타임 에러로 인한 사용자 경험 저하 방지
- 새로운 분석 모드 추가 시 파싱 로직 재사용 가능

**영향 파일**: 
- `lib/response-parser.ts` (기존 로직 통합)
- `app/api/analyze/route.ts`
- `app/api/analyze-flow/route.ts`
- `components/ReportTabs.tsx` (undefined 가드 제거)
- `components/ComparisonReportTabs.tsx` (undefined 가드 제거)

**예상 작업량**: 3-4시간

## Idea 3: TypeScript 빌드 파이프라인 안정화 및 E2E 테스트 신뢰성 확보

**발견 근거**:
- 최신 QA 리포트: "❌ FAIL" on 타입 체크, Playwright 모듈 해석 실패 (TS2307)
- `e2e/` 3개 파일(`api.spec.ts`, `pages.spec.ts`, `settings.spec.ts`)에서 모두 암묵적 `any` 타입 (TS7031)
- 빌드 파이프라인 실패 상황에서도 배포 가능 → CI/CD 검증 미흡
- package.json에 `type-check:e2e` 스크립트 있으나 CI/CD에서 실행 안됨 (확인 필요)
- `tsconfig.json`과 Playwright 설정 불일치 → "로컬 개발과 CI/CD 환경 간 테스트 결과 차이 가능성"

**현재 문제**:
- 빌드 단계에서 타입 체크 실패해도 커밋 가능 → 런타임 에러 발생
- E2E 테스트가 IDE 타입 지원 없음 → 테스트 작성자의 실수 발견 지연
- `@playwright/test` 설치 이력 있으나 tsconfig 미동기화 → 반복 수정 위험
- 로컬 dev 성공 but CI 실패 시나리오 가능 (개발자 경험 악화)

**제안**:
```bash
# 1. tsconfig.json 통합 (현재 분산된 설정 통일)
{
  "compilerOptions": {
    "skipLibCheck": true,
    "types": ["@playwright/test", "node", "jest"]
  },
  "include": ["**/*.ts", "**/*.tsx", "e2e/**/*.ts"]
}

# 2. GitHub Actions 빌드 스텝 강화
validate:
  - npm run lint
  - npm run type-check              # 앱 타입 검사
  - npm run type-check:e2e          # E2E 테스트 타입 검사
  - npm run build                   # 빌드 (실패 시 배포 중단)
  - npx playwright test --reporter=html  # E2E 테스트 실행

# 3. 개발자 로컬 검증 자동화
git/hooks/pre-commit:
  npm run validate (린트 + 타입 + 빌드)
  # 실패 시 커밋 차단
```

**구현 난이도**: Low

**예상 임팩트**: High
- CI/CD 실패율 0으로 (현재 암묵적 에러 노출)
- 개발자 반복 수정 시간 단축 (타입 에러 조기 발견)
- 배포 신뢰도 +40% (타입 안정성 확보)
- Idea 1/2 구현 시 변경 사항 검증 자동화

**영향 파일**: 
- `tsconfig.json` (수정)
- `tsconfig.e2e.json` (통합 또는 삭제)
- `.github/workflows/` (CI/CD 스크립트 강화)
- `e2e/` 모든 파일 (명시적 타입 추가)
- `.husky/pre-commit` (신규 또는 수정)

**예상 작업량**: 4-6시간
