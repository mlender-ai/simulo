### Idea 2: TypeScript 모듈 해석 실패 및 E2E 테스트 타입 안정성 강화

**발견 근거**:
- QA 리포트 CRITICAL 이슈: `@playwright/test` 모듈 TS2307 에러
- `playwright.config.ts`, `e2e/settings.spec.ts` 타입 체크 실패
- E2E 테스트 3개 파일 모두 TS7031 (암묵적 any) 에러 발생
- 최근 커밋 `4fdfc62`: "fix: resolve TypeScript build failure for E2E/Playwright files" 시도했으나 여전히 실패 상태

**현재 문제**:
- `npm run type-check:e2e` 명령이 있지만 `tsconfig.e2e.json` 설정 미완성
- E2E 테스트가 타입 검증 없이 실행되어 런타임 에러 가능성 (자동화 테스트 신뢰성 저하)
- CI/CD 빌드 실패로 자동 배포 파이프라인 차단 (2026-04-15 QA 리포트 기준 ❌ FAIL)
- IDE에서 자동완성 미지원으로 개발 생산성 저하

**제안**:
```
1. tsconfig.e2e.json 별도 설정 정확화
   {
     "extends": "./tsconfig.json",
     "compilerOptions": {
       "skipLibCheck": true,
       "moduleResolution": "node",
       "types": ["@playwright/test", "node"],
       "allowJs": true,
       "checkJs": false,
       "noImplicitAny": false (→ true로 단계적 강화)
     },
     "include": ["e2e/**/*", "playwright.config.ts"],
     "exclude": ["node_modules", ".next"]
   }

2. E2E 테스트 기본 타입 제공 (e2e/fixtures.ts 신규 생성)
   import { test as base, Page, BrowserContext } from '@playwright/test';
   
   type TestFixtures = {
     authenticatedPage: Page;
     dbConnection: any;
   };
   
   export const test = base.extend<TestFixtures>({...});
   export { expect } from '@playwright/test';

3. 모든 E2E 테스트 파일에 명시적 타입 선언
   ✅ e2e/api.spec.ts, e2e/pages.spec.ts, e2e/settings.spec.ts 수정
      test('description', async ({ page }: { page: Page }) => {

4. package.json에 npm postinstall 훅 추가
   "postinstall": "prisma generate && npm run type-check:e2e || true"
```

**구현 난이도**: Low

**예상 임팩트**: High
- CI/CD 빌드 성공률 향상 (현재 ❌ → ✅)
- E2E 테스트 신뢰성 증가 (자동 런타임 에러 감지)
- 개발자 IDE 경험 개선 (자동완성, 타입 힌팅)

**영향 파일**:
- `tsconfig.e2e.json` (수정)
- `playwright.config.ts` (타입 정의 추가)
- `e2e/api.spec.ts`, `e2e/pages.spec.ts`, `e2e/settings.spec.ts` (타입 선언 추가)
- `e2e/fixtures.ts` (신규 생성)
- `package.json` (postinstall 훅 수정)

**예상 작업량**: 1-2일

### Idea 3: JSON 파싱 실패 자동 복구 메커니즘 강화

**발견 근거**:
- Git Log에 JSON 파싱 관련 fix 4건 명시적 기록:
  - `c52252b`: "fix: JSON parse failure when Claude appends trailing prose after closing brace"
  - `7dc4bed`: "fix: comparison analysis JSON truncation and recovery"
  - `3ca6211`: "fix: flow analysis JSON parse failures from truncation"
  - `cee94fa`: "fix: reorder JSON schema so thinkAloud/issues always generated first"
- 최근 핫스팟 파일 `lib/claude.ts`에 이들 파싱 로직 집중
- QA 리포트 HIGH 이슈: "CI가 Secrets 없이도 빌드 가능하도록 fallback 추가" (0e07385 커밋)로 에러 핸들링 패턴 이미 부분 적용

**현재 문제**:
- Claude API 응답이 토큰 제한으로 자르기거나 JSON 뒤에 텍스트를 추가할 때 파싱 실패
- 매번 fix 커밋으로 대응하는 방식 (반복성 높음)
- 사용자는 분석 결과를 받지 못하고 재시도해야 함 (UX 마찰)
- `localStorage quota exceeded` 에러로 대용량 이미지 저장 실패 가능성 (64b314e 참고)

**제안**:
```
1. 응답 파싱 라이브러리 강화 (lib/claude.ts → lib/response-parser.ts)
   - JSON.parse()에서 시작하여 단계적으로 fallback 적용
   - Step 1: 표준 파싱
   - Step 2: 후행 텍스트 제거 후 재파싱
   - Step 3: truncated 감지 시 마지막 완전한 객체만 추출
   - Step 4: 필수 필드 누락 시 기본값 삽입
   - Step 5: 완전 실패 시 명확한 에러 메시지 + 부분 결과 반환

2. 토큰 관리 개선
   - app/api/analyze/route.ts에서 max_tokens를 동적으로 조정
   - 응답 크기 예측 후 미리 여유 남기기
   - 대용량 분석 요청 시 청킹(chunking) 전략 도입

3. 로컬 스토리지 오버플로우 방지
   - IndexedDB 우선 사용 (현재 일부만 적용: 46ba929)
   - 대용량 이미지는 압축 또는 외부 저장소(S3) 활용
   - 저장 전 용량 체크 + 경고

4. 로깅 및 모니터링 강화
   - 파싱 실패 시 상세 로그 기록
   - Claude 응답 크기 / 토큰 사용량 대시보드
   - 실패율 추적으로 패턴 파악
```

**구현 난이도**: Medium

**예상 임팩트**: High
- 사용자 재시도 빈도 감소 (분석 성공률 향상)
- fix 커밋 감소로 코드 안정성 향상
- 프롬프트/API 최적화 근거 데이터 확보

**영향 파일**:
- `lib/claude.ts` (→ `lib/response-parser.ts`로 분리)
- `app/api/analyze/route.ts` (토큰 관리)
- `app/api/analyze-flow/route.ts` (동일 로직 적용)
- `components/ReportTabs.tsx` (에러 UI 개선)
- `lib/storage.ts` (신규: IndexedDB 관리)

**예상 작업량**: 2-3일 (파서 강화 + 모니터링)
