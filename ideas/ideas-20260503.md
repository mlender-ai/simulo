### Idea 1: 프롬프트 버전 관리 레이어 도입 — PromptRegistry 분리

**발견 근거**:
- `lib/prompts.ts`(4회 수정), `lib/claude.ts`(4회 수정), `lib/improvement/opusGenerator.ts`(3회 수정)가 모두 핫스팟 상위권에 동시 진입
- `fix(improvement): rewrite generator prompt for surgical fixes` 커밋이 별도로 존재하는 것은 프롬프트 품질 문제가 코드 배포 사이클에 직접 묶여 있음을 의미
- `9e12145` 커밋에서 프롬프트를 "전면 재작성"했음에도 이후 race condition fix(`41a89db`, `fd360ff`)가 연속 발생 — 프롬프트 변경이 런타임 동작에 예측 불가능하게 영향을 미치고 있음

**현재 문제**:
- 프롬프트 문자열이 `lib/prompts.ts`와 `opusGenerator.ts`에 하드코딩되어 있어, 프롬프트 한 줄 수정에도 전체 빌드·배포 사이클이 필요함
- 프롬프트 변경 이력이 git diff로만 추적 가능하여 "어떤 프롬프트 버전에서 어떤 품질의 결과가 나왔는지" 재현이 불가능
- 일반 모드 vs 야핏무브 모드, 도메인별 포커스 영역 등 분기 조건이 코드 안에 섞여 `lib/claude.ts`의 893줄 규모를 키우는 주 원인이 됨

**제안**:
```
lib/prompts/
├── registry.ts          # PromptRegistry — 버전·태그 기반 조회 인터페이스
├── versions/
│   ├── improvement-v1.ts
│   ├── improvement-v2.ts  # 현재 "surgical fix" 버전
│   └── analysis-v1.ts
├── templates/
│   ├── base.ts           # 공통 변수 치환 유틸
│   └── domain.ts         # 도메인별 포커스 영역 상수
└── types.ts              # PromptVersion, PromptTag 타입
```
- `PromptRegistry.get(name, version?)` 인터페이스로 프롬프트를 코드에서 분리
- 각 프롬프트 버전에 `schemaVersion` 필드를 부여해 `outputSchema.ts`의 스키마 검증과 연결
- 단기적으로는 기존 하드코딩 문자열을 `versions/` 파일로 이동하는 것만으로 즉시 효과 발생

**구현 난이도**: Medium
**예상 임팩트**: High
**영향 파일**:
- `lib/prompts.ts`
- `lib/claude.ts`
- `lib/improvement/opusGenerator.ts`
- `app/api/analyze/route.ts`
- `app/api/generate-improvement/route.ts`

**예상 작업량**: 2~3일

### Idea 2: 공유 경험 통합 레이어 — ShareExportPanel 중심의 공유 플로우 일원화

**발견 근거**:
- 최근 7일 내 공유/내보내기 관련 커밋이 파편적으로 산재: 탭 딥링크(`72b92a2`), 클립보드 복사 버튼(`590d8e4`), CSV 내보내기(`f2a515f`)
- `app/share/[id]/page.tsx`가 앱 구조에 존재하고 `ShareExportPanel.tsx` 컴포넌트도 있음에도 불구하고, 실제 공유 기능(링크 복사, CSV, PNG 저장, 클립보드)이 각기 다른 컴포넌트에 분산 구현되어 있음
- 이전 아이디어(04/28~04/29)에서 히트맵·대시보드 영역을 다뤘으나 공유 플로우는 미검토 영역

**현재 문제**:
- 이슈 탭 클립보드 복사(`ReportTabs` 또는 `UsabilityReportTabs` 내부), PNG 저장(`ImprovementPanel` 내부), CSV 내보내기(`ShareExportPanel`), 딥링크 URL(`?tab=` 쿼리파람)이 각기 다른 컴포넌트에 구현되어 공유 진입점이 4곳으로 분산
- 사용자가 분석 결과를 팀원에게 공유할 때 "어디서 무엇을 내보낼 수 있는지" 발견 불가 — UX 마찰의 핵심 원인
- `app/share/[id]/page.tsx`의 공유 페이지가 현재 얼마나 활용되는지 추적 불가 (공유 이벤트 계측 부재)

**제안**:
```
공유 플로우 통합 3단계:

1단계 — ShareExportPanel 기능 통합 (1일)
   - 클립보드 복사, PNG 저장, CSV 내보내기, 딥링크 생성을
     ShareExportPanel.tsx 단일 진입점으로 집결
   - 각 컴포넌트에 흩어진 내보내기 로직 → useShareExport() 훅으로 추출

2단계 — 공유 컨텍스트 상태 관리 (1일)
   - 현재 활성 탭, 적용된 필터(severity), 선택된 존 등
     "현재 뷰 상태"를 share URL에 직렬화하는 ShareStateEncoder 도입
   - ?tab=issues&severity=critical&zone=header 형태의 딥링크 지원

3단계 — 공유 이벤트 계측 (0.5일)
   - 어떤 형식(링크/CSV/PNG)이 실제로 사용되는지
     localStorage 기반 경량 이벤트 로깅으로 우선 추적
```

**구현 난이도**: Medium
**예상 임팩트**: Medium
**영향 파일**:
- `components/ShareExportPanel.tsx`
- `components/ReportTabs.tsx`

### Idea 3: CI 타입 안전성 게이트 — TypeScript 빌드 실패 자동 차단 파이프라인

**발견 근거**:
- QA 리포트(2026-04-15)에서 빌드 `⚠️ FAIL`, 타입 체크 `❌ FAIL`이 동시에 기록됨에도 이후 7일간 관련 fix 커밋이 없음 — 빌드 실패가 개발 흐름을 차단하지 못하고 있음
- `lib/claude.ts`(893줄), `components/InputSection.tsx`(824줄)가 핫스팟 상위에서 반복 수정되고 있으나, 대용량 파일에 대한 자동화된 복잡도 경고가 없어 분리 시점을 놓치고 있음
- `@playwright/test` 모듈 누락(TS2307), 암묵적 `any` 타입(TS7031)이 QA에서 반복 지적되는데, 이는 로컬에서 타입 체크를 실행하지 않고 커밋이 이루어지고 있음을 시사
- fix 비율 27%는 "사후 수정" 비용이 여전히 높다는 신호

**현재 문제**:
- `npm run build`와 `tsc --noEmit`이 CI에서 직렬로 검증되지 않아, 타입 오류가 있는 코드가 main 브랜치에 머지됨
- E2E 테스트 3개 파일 전체에 타입 오류가 있는 상태에서 `E2E ✅ PASS`가 리포트되는 모순 — 테스트 신뢰성 자체가 훼손됨
- `skipLibCheck: true` 여부가 불명확하여 로컬과 CI 환경 간 타입 체크 결과가 달라질 수 있음

**제안**:
```
3-레이어 CI 타입 안전성 게이트:

Layer 1 — Pre-commit 훅 (0.5일)
   # .husky/pre-commit
   npx tsc --noEmit --skipLibCheck false
   # 타입 오류 시 커밋 차단 — 근본 원인 차단

Layer 2 — 파일 복잡도 경고 (0.5일)
   # CI 스크립트에 추가
   find . -name "*.tsx" -o -name "*.ts" | xargs wc -l | awk '$1 > 500'
   # 500줄 초과 파일 목록을 PR 코멘트로 자동 게시
   # (차단 아닌 경고 — 분리 시점 시각화)

Layer 3 — tsconfig 환경 통일 (1일)
   # tsconfig.json 수정
   {
     "compilerOptions": {
       "strict": true,
       "skipLibCheck": false,
       "types": ["@playwright/test", "node"]
     }
   }
   # playwright.config.ts를 별도 tsconfig.e2e.json으로 분리해
   # 앱 빌드 타입 체크와 E2E 타입 체크를 독립적으로 관리
```
- 즉시 효과: 타입 오류 커밋 사전 차단 → fix 비율 감소 기대
- 중기 효과: 복잡도 경고 → `InputSection.tsx`, `claude.ts` 분리 타이밍 자연스럽게 유도

**구현 난이도**: Low
**예상 임팩트**: High
**영향 파일**:
- `tsconfig.json`
- `playwright.config.ts`
- `e2e/settings.spec.ts`
- `e2e/api.spec.ts`
- `e2e/pages.spec.ts`
- `.husky/pre-commit` (신규)
- `package.json`

**예상 작업량**: 1.5~2일
