### Idea 1: `generate-improvement` 파이프라인 컨텍스트 계층 분리 — API 키·모델 설정의 단일 진입점화

**발견 근거**:
- `81da356` (generate-improvement API가 사용자 로컬 API 키를 무시하는 버그)와 `3ad4a8f` (dashboard insights button — pass user API key to insights route)가 각각 별도 fix로 등장 — 동일한 "API 키 전달 누락" 패턴이 라우트마다 반복 발생
- `ImprovementPanel.tsx`가 5회 변경·상태 머신 재설계까지 거쳤음에도 API 연동 계층의 버그가 재발한 것은, 컴포넌트 상태 관리와 API 호출 컨텍스트(키·모델 설정)가 서로 다른 레이어에 분산되어 있기 때문으로 진단됨
- 이전 아이디어(ideas-20260505)에서 ImprovementPanel 상태 관리 자체는 다뤘으나, API 호출 컨텍스트의 단일 진입점 부재는 이번 주 새로 확인된 패턴

**현재 문제**:
- `app/api/generate-improvement/route.ts`, `app/api/analyze/route.ts`, `app/api/dashboard/stats/route.ts` 등 각 라우트가 독립적으로 사용자 API 키를 요청 헤더/바디에서 추출하는 로직을 중복 구현
- API 키가 누락될 경우 라우트마다 다른 방식으로 폴백하거나 조용히 실패(silent failure)하여 사용자는 결과 이상을 보고서야 문제를 인지함
- `lib/claude.ts`(893줄)가 API 클라이언트·프롬프트·파싱을 모두 보유한 모놀리스 구조이므로, 키 주입 지점이 어디인지 추적 자체가 어려움

**제안**:
1. `lib/api-context.ts` 신규 생성 — 요청 객체에서 `x-user-api-key` 헤더를 추출하고, 없을 경우 환경 변수 폴백하며, 모델 설정까지 묶어 `ApiContext` 타입으로 반환하는 단일 헬퍼
2. 모든 API 라우트가 최상단에서 `resolveApiContext(req)` 한 번만 호출 후 하위로 전달하는 컨벤션 표준화
3. `lib/claude.ts`를 QA 리포트 리팩토링 제안 구조대로 `ai-client.ts` / `prompt-builder.ts` / `response-parser.ts`로 분리하되, `ai-client.ts`가 `ApiContext`를 유일한 설정 소스로 받도록 설계
4. API 키 미전달 시 라우트 레벨에서 즉시 400 응답 + 사용자 향 에러 메시지 통일

```
lib/
├── api-context.ts          # resolveApiContext() 단일 헬퍼 (신규)
├── claude/
│   ├── ai-client.ts        # ApiContext 주입받아 Claude 호출
│   ├── prompt-builder.ts
│   ├── response-parser.ts
│   └── types.ts
```

**구현 난이도**: Medium  
**예상 임팩트**: High  
**영향 파일**:
- `app/api/generate-improvement/route.ts`
- `app/api/analyze/route.ts`
- `app/api/dashboard/stats/route.ts`
- `app/api/report/route.ts`
- `lib/claude.ts` (분리 대상)
- `lib/improvement/opusGenerator.ts`

**예상 작업량**: 2~3일

### Idea 2: `InputSection.tsx` 입력 단계 분리 — 단계형 분석 설정 위저드(Stepper)로 전환

**발견 근거**:
- `components/InputSection.tsx`가 이번 주 핫스팟 공동 2위(4회 변경, 824줄)이며, 이번 주에만 collapsible 폼 섹션(`9bc2642`), 제품 설명·참조 이미지 입력(`3df6de8`→`3ad4a8f` 범위), 반응형 레이아웃(`aa72d6e`) 변경이 중첩됨
- 기능이 누적되며 단일 컴포넌트가 "미디어 업로드 + 파라미터 설정 + 프리셋 + 제품 설명 + 참조 이미지"를 동시에 렌더링하는 구조로 팽창 중
- QA 리포트에서도 `InputSection.tsx`(824줄)를 SRP 위반 및 테스트 작성 어려움의 원인으로 명시

**현재 문제**:
- 사용자가 분석을 시작할 때 모든 설정 항목이 한 화면에 펼쳐져 "어디서부터 채워야 하는가"의 인지 부하가 높음
- collapsible 섹션으로 임시 해소했으나, 어떤 입력이 필수/선택인지 시각적 구분이 없어 불완전한 상태로 분석 요청이 발생할 가능성 존재
- 824줄 단일 파일 구조로 인해 신규 입력 타입(영상, OCR, Figma 등)이 추가될 때마다 동일 파일이 변경점 충돌 핫스팟이 됨

**제안**:
1. `InputSection.tsx`를 3단계 Stepper 컴포넌트로 재설계
   - **Step 1 — 분석 대상**: 미디어 업로드(이미지/영상/URL/Figma) 단독 표시
   - **Step 2 — 분석 설정**: 모드·도메인·페르소나·프리셋 선택
   - **Step 3 — 컨텍스트 보강(선택)**: 제품 설명, 참조 이미지, 경쟁 화면 첨부
2. 각 Step을 독립 컴포넌트로 분리하여 단위 테스트 가능하도록 구성
3. Step 간 상태는 `useInputWizard` 훅으로 단일 관리, URL 쿼리 파라미터로 Step 위치를 유지해 새로고침 시 이탈 방지
4. Step 1 완료 시점부터 "분석 시작" CTA 활성화 — 선택 항목 미입력을 이유로 흐름이 막히지 않도록

```
components/input/
├── InputWizard.tsx         # 스텝 컨테이너 (리팩토링된 InputSection)
├── steps/
│   ├── Step1MediaUpload.tsx
│   ├── Step2AnalysisConfig.tsx
│   └── Step3ContextEnrich.tsx
└── hooks/
    └── useInputWizard.ts
```

**구현 난이도**: Medium  
**예상 임팩트**: High  
**영향 파일**:
- `components/InputSection.tsx` (분해 대상)
- `app/page.tsx`
- `app/analyze/page.tsx`
- `components/MediaUploader.tsx`
- `components/OCRReviewModal.tsx`

**예상 작업량**: 3~4일

### Idea 3: 분석 리포트 팀 공유 워크스페이스 — 코멘트·액션 아이템 레이어 추가

**발견 근거**:
- `app/share/[id]/page.tsx`와 `components/ShareExportPanel.tsx`가 앱 구조에 이미 존재하나, 이번 주 Git 로그에서 share 관련 커밋이 전혀 없음 — 공유 기능이 구현되어 있으나 개선 없이 방치된 상태
- `feat: 이슈 탭 클립보드 복사 버튼 추가(#33)`, `feat: tab URL deep-link support via ?tab=` 등 "결과를 외부로 꺼내는" 수요가 이슈 티켓과 커밋으로 반복 확인됨 — 사용자가 분석 결과를 팀에 전달하려 하지만 현재 도구가 링크 복사 수준에 머물러 있음
- multi-persona Think Aloud(`3994142`), 시안 비교 탭(`1015413`) 등 멀티-관점 분석 기능이 추가되면서 "여러 사람이 함께 검토"하는 시나리오의 필요성이 높아졌으나 협업 레이어가 없음

**현재 문제**:
- 공유 링크(`/share/[id]`)를 받은 팀원이 리포트를 읽기만 할 수 있고, 이슈에 코멘트를 달거나 담당자를 지정하거나 수정 완료 여부를 표시하는 방법이 없음
- 결과적으로 Simulo 리포트 → Slack/Notion 복붙 → Jira 티켓 생성의 수동 3단계가 발생하여, 분석 인사이트가 실무 태스크로 연결되는 경로에 마찰이 집중됨
- `components/IssuesTab.tsx`가 이번 주 핫스팟(2회)에 포함되어 있으며, 이슈 카드 접기·복사 기능 추가가 이미 진행 중 — 이슈 카드가 실무 액션의 단위로 발전할 준비가 된 시점

**제안**:
1. 이슈 카드에 **액션 아이템 레이어** 추가
   - 각 이슈에 "할 일로 등록" 토글 → 담당자(이메일) + 기한 입력 → `/api/history` 또는 신규 `/api/actions` 엔드포인트에 저장
   - 공유 링크 수신자도 액션 등록 가능(인증 없이 닉네임만 입력)
2. 리포트 상단에 **팀 진행 현황 바** 추가 — "총 N개 이슈 중 M개 완료" 시각화, 완료 이슈는 카드 색상 변경으로 구분
3. **Notion/Linear 내보내기** 버튼 추가 — 이슈 목록을 마크다운 체크리스트(Notion) 또는 JSON(Linear API) 형식으로 원클릭 내보내기
4. 공유 페이지(`/share/[id]`)에 위 기능 통합하여 링크 하나로 검토·코멘트·태스크 할당까지 완결

```
app/share/[id]/page.tsx         # 액션 레이어 통합
app/api/actions/route.ts        # 신규 — 액션 아이템 CRUD
components/report/IssuesTab.tsx # 액션 등록 UI 추가
components/ActionStatusBar.tsx  # 신규 — 진행 현황 바
lib/export/
├── notion.ts                   # Notion 마크다운 변환
└── linear.ts                   # Linear JSON 변환
```

**구현 난이도**: High  
**예상 임팩트**: High  
**영향 파일**:
- `app/share/[id]/page.tsx`
- `components/report/IssuesTab.tsx`
- `components/ShareExportPanel.tsx`
- `app/api/history/route.ts`
- `app/api/actions/route.ts` (신규)

**예상 작업량**: 5~7일
