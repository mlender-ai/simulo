### Idea 1: Claude 응답 보호층 도입 — JSON 파싱 실패 근절

**발견 근거**: 
- Git 히스토리에서 JSON 파싱 관련 fix 4건 확인:
  - `b7dc4bed` fix: JSON parse failure when Claude appends trailing prose
  - `7ac2c2a` fix: ensure all required fields exist after Claude response parse
  - `7dc4bed` fix: comparison analysis JSON truncation and recovery
  - `cea94fa` fix: reorder JSON schema so thinkAloud/issues always generated first
- 현재 `lib/claude.ts` (893줄)에서 응답 처리 로직이 단순 try-catch 수준
- QA 리포트에서 "max_tokens 트렁케이션으로 인한 500 에러" 명시

**현재 문제**: 
- Claude가 JSON 종료 후 추가 텍스트를 붙이거나, 응답이 중간에 끝나면 파싱 실패
- 각 API 엔드포인트(`/api/analyze`, `/api/analyze-flow`, `/api/report`)에서 서로 다른 방식으로 에러 처리 중
- 사용자는 분석 중 갑자기 500 에러 또는 빈 리포트 경험

**제안**: 
1. **응답 정규화 모듈 신규 작성** (`lib/claude/response-sanitizer.ts`):
   - JSON 추출 로직: 첫 `{`부터 마지막 `}`까지만 추출 (정규식)
   - 필드 검증: 스키마 정의 후 누락된 필드는 기본값으로 채우기
   - 토큰 예측: 프롬프트 길이 기반 max_tokens 자동 계산 (안전 마진 20% 확보)

2. **프롬프트 템플릿 표준화** (`lib/claude/prompt-builder.ts`):
   - 모든 분석 요청에 "반드시 유효한 JSON으로 끝낼 것" 명시
   - JSON 스키마를 프롬프트에 포함하여 Claude의 정확도 향상
   - 응답이 길어질 경우 사전에 분석 범위 축소

3. **폴백 전략**:
   - 파싱 실패 시 1회 재시도 (같은 프롬프트로)
   - 재시도 후에도 실패하면 부분 데이터라도 반환 (null 대신 기본 리포트 구조)

**구현 난이도**: Medium
**예상 임팩트**: High (현재 주요 에러 원인 5건 모두 해결)
**영향 파일**: 
- `lib/claude.ts` (리팩토링)
- `app/api/analyze/route.ts`
- `app/api/analyze-flow/route.ts`
- `app/api/report/route.ts`
- `lib/storage.ts` (에러 복구 로직 추가)

**예상 작업량**: 3-4일

### Idea 2: E2E 테스트 타입 안정성 강화 — 빌드 파이프라인 차단 해제

**발견 근거**:
- QA 리포트 "CRITICAL" 섹션: `@playwright/test` 모듈 해석 실패 (TS2307)
- 최신 커밋 `4fdfc62` "fix: resolve TypeScript build failure for E2E/Playwright files"에서 부분 해결 시도했으나 불완전
- 3개 E2E 파일 모두 암묵적 `any` 타입 (TS7031) 발생
- 현재 `tsconfig.json`과 `playwright.config.ts`가 동기화 미흡
- CI/CD: `type-check` 작업 FAIL 상태이므로 배포 차단 가능성

**현재 문제**:
- 개발자가 E2E 테스트 작성/수정 시 IDE에서 타입 지원 받지 못함
- CI 파이프라인에서 빌드 성공해도 배포 전 타입체크 실패로 인한 지연
- 신규 테스트 추가 시마다 같은 에러 반복 (수동 수정 필요)

**제안**:
1. **TypeScript 설정 통합** (`tsconfig.json` 및 `tsconfig.e2e.json` 신규):
   ```json
   {
     "extends": "./tsconfig.json",
     "compilerOptions": {
       "skipLibCheck": true,
       "types": ["@playwright/test", "node", "jest"]
     },
     "include": ["e2e/**/*.ts", "playwright.config.ts"]
   }
   ```
   - 현재 `npm run type-check:e2e` 스크립트 이미 존재 → 이를 CI에 추가

2. **E2E 테스트 기본 구조 자동화** (`e2e/fixtures.ts` 신규):
   ```typescript
   import { test as base, Page } from '@playwright/test';
   
   type TestFixtures = {
     authenticatedPage: Page;
   };
   
   export const test = base.extend<TestFixtures>({
     authenticatedPage: async ({ page }, use) => {
       // 공통 셋업 로직
       await use(page);
     },
   });
   ```
   - 기존 `api.spec.ts`, `pages.spec.ts`, `settings.spec.ts`에 적용

3. **package.json 의존성 점검**:
   - `@playwright/test@^1.59.1` 설치 확인 (이미 등재됨)
   - `@types/node` 포함 확인 및 버전 업그레이드

4. **CI 워크플로우 개선** (`.github/workflows/auto-qa-ideas.yml`):
   ```yaml
   - name: Type Check E2E
     run: npm run type-check:e2e
   ```

**구현 난이도**: Low
**예상 임팩트**: Medium (개발 생산성 향상, CI 안정성 증대)
**영향 파일**:
- `tsconfig.json`
- `tsconfig.e2e.json` (신규)
- `e2e/fixtures.ts` (신규)
- `e2e/api.spec.ts`, `e2e/pages.spec.ts`, `e2e/settings.spec.ts` (타입 추가)
- `playwright.config.ts`
- `.github/workflows/auto-qa-ideas.yml`

**예상 작업량**: 1-2일

### Idea 3: 대용량 파일 분리 — 유지보수성 및 테스트 커버리지 향상

**발견 근거**:
- 핫스팟 파일 분석:
  - `lib/claude.ts`: 893줄 (AI 통신 + 프롬프트 생성 + 응답 파싱 혼재)
  - `components/ReportTabs.tsx`: 811줄 (4개 탭의 렌더링 로직 일괄 처리)
  - `components/InputSection.tsx`: 824줄 (파일 업로드 + 파라미터 + 프리셋 혼재)
- QA 리포트에서 "단일 책임 원칙(SRP) 위반, 유지보수성 저하" 명시
- Git 히스토리에서 이 3개 파일의 수정이 자주 발생 → 변경 부작용 위험 높음

**현재 문제**:
- 한 파일에 여러 책임이 섞여있어 단위 테스트 작성 어려움
- 특정 기능 버그 수정 시 다른 기능에 영향 가능성
- 새 팀 멤버의 온보딩 시간 증가
- 번들 사이즈 최적화 어려움 (코드 스플리팅 불가)

**제안**:

1. **`lib/claude.ts` (893줄) → 모듈화 구조**:
   ```
   lib/claude/
   ├── index.ts          (공개 인터페이스)
   ├── client.ts         (Claude API 호출 로직)
   ├── prompt-builder.ts (프롬프트 생성, 토큰 계산)
   ├── response-parser.ts (JSON 파싱, 필드 검증)
   ├── types.ts          (AnalysisRequest, AnalysisResponse 등)
   └── constants.ts      (프롬프트 템플릿, 스키마 정의)
   ```
   - 각 모듈 < 300줄로 유지
   - 단위 테스트 추가 용이

2. **`components/InputSection.tsx` (824줄) → 컴포넌트 분해**:
   ```
   components/input/
   ├── InputSection.tsx      (상태 관리, 컨테이너)
   ├── FileUploader.tsx      (파일 드래그앤드롭)
   ├── ParameterControl.tsx  (설정값 입력 폼)
   ├── PresetSelector.tsx    (프리셋 선택)
   └── hooks/
       └── useInputState.ts  (상태 로직 추출)
   ```

3. **`components/ReportTabs.tsx` (811줄) → 탭별 분리**:
   ```
   components/report/
   ├── ReportTabs.tsx        (탭 네비게이션, 기본 구조)
   ├── tabs/
   │   ├── OverviewTab.tsx   (요약 정보, 점수)
   │   ├── DetailTab.tsx     (상세 분석 이슈)
   │   ├── HeatmapTab.tsx    (히트맵 시각화)
   │   └── ComparisonTab.tsx (비교 분석)
   └── shared/
       └── IssueCard.tsx     (공통 이슈 카드 컴포넌트)
   ```

4. **리팩토링 검증**:
   - 각 모듈/컴포넌트별 Playwright E2E 테스트 추가
   - 기존 기능 동작 확인 후 머지

**구현 난이도**: High
**예상 임팩트**: Medium (장기적 유지보수성, 테스트 커버리지 향상)
**영향 파일**:
- `lib/claude.ts` → `lib/claude/*` (신규 구조)
- `components/InputSection.tsx` → `components/input/*` (신규 구조)
- `components/ReportTabs.tsx` → `components/report/*` (신규 구조)
- `app/api/analyze/route.ts`, `app/api/analyze-flow/route.ts` (import 경로 수정)

**예상 작업량**: 5-6일 (테스트 포함)
