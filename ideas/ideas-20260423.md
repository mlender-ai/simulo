### Idea 1: 분석 모드 플러그인 레지스트리 — 핸들러 확장 아키텍처 표준화

**발견 근거**:
- `app/api/analyze/route.ts`가 7일 간 5회 변경(전체 핫스팟 1위)
- 최근 `handlers/` 디렉터리 분리(`figma.ts`, `comparison.ts`, `flow.ts`, `image.ts`)가 이뤄졌으나, 신규 모드(영상, OCR, code-to-figma)는 별도 라우트(`upload-video`, `ocr-extract`, `code-to-figma`)로 파편화
- 향후 분석 모드가 추가될 때마다 `route.ts` 분기 로직이 재증가하는 패턴이 반복될 구조

**현재 문제**:
- `route.ts`가 모드 판별 + 핸들러 위임 + 에러 처리를 혼재 — 새 모드 추가 시 진입점 파일이 다시 비대화
- 각 분석 모드의 입력 스키마·응답 형식·에러 처리가 핸들러마다 제각각이어서 일관된 테스트 작성이 불가
- `handlers/types.ts`가 존재하지만 실제 타입 공유 범위가 불명확

**제안**:
분석 모드를 **자기 서술적(self-describing) 플러그인**으로 등록하는 레지스트리 패턴 도입

```
lib/analyze/
├── registry.ts          # 모드 등록 및 디스패치 엔진
├── modes/
│   ├── image.mode.ts    # { id, inputSchema, handler, outputSchema }
│   ├── flow.mode.ts
│   ├── comparison.mode.ts
│   ├── figma.mode.ts
│   ├── video.mode.ts    # 신규 통합
│   └── ocr.mode.ts      # 신규 통합
└── shared/
    ├── base-handler.ts  # 공통 에러 처리·로깅 래퍼
    └── types.ts         # 모드 인터페이스 정의
```

`route.ts`는 `registry.dispatch(mode, payload)` 단일 호출로 축소. 신규 모드는 `modes/` 디렉터리에 파일 추가만으로 등록 완료.

**구현 난이도**: Medium
**예상 임팩트**: High
**영향 파일**:
- `app/api/analyze/route.ts`
- `app/api/analyze/handlers/figma.ts`
- `app/api/analyze/handlers/comparison.ts`
- `app/api/analyze/handlers/flow.ts`
- `app/api/analyze/handlers/image.ts`
- `app/api/upload-video/route.ts`
- `app/api/ocr-extract/route.ts`
- `lib/claude.ts`

**예상 작업량**: 3~4일

### Idea 2: 분석 결과 → 실무 연결 허브 — "Action Export" 패널

**발견 근거**:
- 이번 주 신규 기능 5개(3-Pass OCR, 영상 분석, code-to-Figma 역방향, 비교 분석 체크박스, 개선 생성 안정화) 중 **결과 활용 관련은 0개**
- 앱 구조상 `app/share/[id]`, `app/report/[id]`, `app/history` 라우트와 `components/ShareExportPanel.tsx`가 존재하나 Git log에서 해당 파일의 변경 이력이 전무 — 기능 정체 확인
- `app/api/export/docx/[id]`, `app/api/export/pdf/[id]` API가 존재하나 UI 레벨에서의 연결이 `ShareExportPanel` 단일 컴포넌트에만 의존

**현재 문제**:
- 사용자가 분석 완료 후 결과를 팀에 공유하거나 티켓으로 전환하는 경로가 PDF/DOCX 다운로드 외에 없음
- `ImprovementPanel.tsx`에서 생성된 개선 제안이 별도 액션(Jira 티켓, Notion 페이지, GitHub Issue)과 연결되지 않아 분석 → 실행 사이의 컨텍스트 스위칭 비용이 높음
- `app/history/page.tsx`가 단순 목록 조회에 머물러, 반복 분석 간 트렌드나 개선 이행률을 확인할 수 없음

**제안**:
분석 리포트 하단에 **"다음 액션" 허브 패널** 신설

```
components/ActionExportHub/
├── ActionExportHub.tsx       # 컨테이너 (ReportTabs 하단 고정)
├── ExportTargets/
│   ├── MarkdownExport.tsx    # 개선 제안을 Markdown으로 복사
│   ├── JiraTicketDraft.tsx   # 이슈 제목+설명 자동 생성 (클립보드)
│   └── NotionBlockExport.tsx # Notion API 연동 (선택적)
└── FollowUpTracker.tsx       # 개선 제안별 Done/In Progress 체크박스
                               # → history에 이행률 저장
```

1단계(즉시 구현 가능): Markdown 복사 + Jira 드래프트 자동 생성 (외부 API 불필요)
2단계: `history` 페이지에 "제안 이행률" 집계 뷰 추가

**구현 난이도**: Low (1단계) / Medium (2단계)
**예상 임팩트**: High
**영향 파일**:
- `components/ImprovementPanel.tsx`

- `components/ReportTabs.tsx`

**예상 작업량**: 1단계 1~2일 / 2단계 추가 2일

### Idea 3: OCR + 영상 분석 입력 품질 게이트 — Pre-flight Validation 레이어

**발견 근거**:
- 이번 주 `feat: 3-Pass OCR 보정 파이프라인 신설`과 `feat: 영상 업로드 + 프레임 분석` 두 신규 입력 경로가 동시에 추가됨
- `components/OCRReviewModal.tsx`와 `components/MediaUploader.tsx`가 신설됐으나, 입력 품질 검증 로직이 클라이언트 컴포넌트 안에 산재할 가능성이 높음
- 영상 프레임 분석은 파일 크기·해상도·프레임 수에 따라 Claude API 비용과 응답 품질이 크게 달라지지만, 사전 검증 레이어가 별도로 존재하지 않음 (QA 리포트에 관련 이슈 부재 = 아직 실사용 전 또는 에러 미기록)

**현재 문제**:
- 저화질 이미지·압축된 영상·OCR 인식률 낮은 스크린샷이 그대로 Claude API에 전달될 경우, 분석 품질 저하가 사용자에게 에러가 아닌 **잘못된 결과**로 나타남 (silent failure)
- `api/upload-video/route.ts`와 `api/ocr-extract/route.ts`가 독립 라우트로 존재해 입력 품질 기준이 라우트마다 따로 구현되거나 누락될 위험
- 3-Pass OCR 파이프라인의 보정 비용(API 호출 3회)이 낮은 품질 입력에도 동일하게 적용되어 불필요한 지연 발생

**제안**:
모든 멀티모달 입력에 공통 적용되는 **Pre-flight Validation 미들웨어** 도입

```
lib/input-validation/
├── preflight.ts              # 공통 게이트 인터페이스
├── validators/
│   ├── image.validator.ts    # 해상도·파일크기·포맷 체크
│   │                          # → 저화질 시 샤프닝 힌트 또는 업로드 거부
│   ├── video.validator.ts    # 길이·프레임레이트·해상도 체크
│   │                          # → 적정 프레임 추출 구간 자동 제안
│   └── ocr.validator.ts      # 텍스트 밀도·명암비 예측 점수
│                              # → OCR 신뢰도 낮으면 3-Pass 자동 활성화
└── PreflightReport.ts        # { passed, warnings, suggestions } 구조 반환
```

- 클라이언트: `MediaUploader`에서 업로드 전 **품질 점수 프리뷰** 표시 ("이 영상은 프레임 추출 품질이 낮을 수 있습니다 — 해상도 720p 이상을 권장합니다")
- 서버: 각 API 라우트 진입 시 `preflight()` 통과 여부를 로깅하여 향후 `.claude/memory/patterns.md` 학습 데이터로 축적

**구현 난이도**: Medium
**예상 임팩트**: Medium
**영향 파일**:
- `app/api/upload-video/route.ts`
- `app/api/ocr-extract/route.ts`
- `app/api/analyze/handlers/image.ts`
- `components/MediaUploader.tsx`
- `components/OCRReviewModal.tsx`
- `components/InputSection.tsx`

**예상 작업량**: 2~3일
