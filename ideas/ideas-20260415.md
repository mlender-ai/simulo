## Idea 1: 안정성 강화를 위한 Claude API 응답 파싱 파이프라인 표준화

**발견 근거**:
- Git 로그에서 반복되는 파싱 에러:
  - `fix: JSON parse failure when Claude appends trailing prose after closing brace` (4fdfc62)
  - `fix: ensure all required fields exist after Claude response parse` (c52253b)
  - `fix: JSON parse failures from truncation` (3210914)
  - `fix: comparison analysis JSON truncation and recovery` (7dc4bed)
- 최신 QA 리포트에서 타입 체크 실패(TypeScript) 및 빌드 실패 기록
- lib/claude.ts (893줄)가 최빈 변경 파일로, 응답 파싱 로직의 복잡성 증가

**현재 문제**:
- Claude의 응답이 JSON 닫는 괄호 후 추가 텍스트를 포함할 때 파싱 실패
- 필드 누락/불완전한 응답에 대한 재시도 로직 부재
- 응답 길이 제한(max_tokens)으로 인한 자동 절단 → 데이터 손실
- ReportTabs.tsx, comparison analysis 등 여러 컴포넌트에서 undefined 방어 코드 산재

**제안**:
1. **응답 파싱 계층 분리** (`lib/claude/response-parser.ts` 신규):
   - 토큰화된 JSON 추출 유틸 (closing brace 이후 문자열 제거)
   - 필드 검증 & 자동 복구 로직 (null → 기본값 대체)
   - 재시도 메커니즘 (truncation 감지 시 max_tokens 동적 조정)

2. **스키마 검증 강화**:
   ```typescript
   // lib/claude/schema-validator.ts
   export const validateAnalysisResponse = (raw: unknown): AnalysisResult => {
     const schema = z.object({
       thinkAloud: z.string(),
       issues: z.array(issueSchema).default([]),
       improvements: z.array(improvementSchema).default([])
     });
     return schema.parse(raw);
   };
   ```

3. **테스트 케이스 추가** (e2e/api.spec.ts 확장):
   - 잘린 JSON 응답 처리
   - 필드 누락 응답 처리
   - 추가 텍스트가 포함된 응답 처리

**구현 난이도**: Medium

**예상 임팩트**: High
- fix 커밋 비율 55% → 30% 이상 감소 예상
- 사용자 경험: "분석 실패" 에러 메시지 제거 → 자동 복구
- 온보딩 단계의 신뢰도 향상

**영향 파일**:
- `lib/claude.ts` (리팩토링)
- `lib/claude/response-parser.ts` (신규)
- `lib/claude/schema-validator.ts` (신규)
- `app/api/analyze/route.ts` (파싱 로직 위임)
- `app/api/analyze-flow/route.ts` (파싱 로직 위임)
- `components/ReportTabs.tsx` (undefined 방어 코드 제거 가능)
- `e2e/api.spec.ts` (테스트 케이스 추가)

**예상 작업량**: 3-4일 (파싱 로직 + 테스트 + 통합)

## Idea 2: 대용량 컴포넌트 분리를 통한 테스트 용이성 & 변경 사이클 개선

**발견 근거**:
- 핫스팟 파일 분석:
  - `lib/claude.ts`: 893줄 (최다 변경, 난잡한 의존성)
  - `components/ReportTabs.tsx`: 811줄 (3회 수정, UI 논리 혼재)
  - `components/InputSection.tsx`: 824줄 (복잡한 상태 관리)
- 최신 QA 리포트에서 E2E 테스트 타입 에러: `page` 매개변수 암묵적 any (TS7031)
- 커밋 히스토리: 대용량 파일의 부분 수정이 전체 컴포넌트 재컴파일 유발

**현재 문제**:
- 단일 파일에서 여러 책임 담당 → 변경 시 부작용 위험 (예: ReportTabs 수정 시 undefined 가드 코드 누락)
- E2E 테스트의 타입 안정성 부족 → 자동 완성 미지원, 런타임 에러 발생 가능
- 대용량 파일로 인한 IDE 성능 저하, 코드 리뷰 난이도 증가

**제안**:
1. **ReportTabs.tsx (811줄) → 탭 컴포넌트 분리**:
   ```
   components/report/
   ├── ReportTabs.tsx (탭 라우팅만)
   ├── tabs/
   │   ├── OverviewTab.tsx (지표, 요약)
   │   ├── IssuesTab.tsx (문제점 목록)
   │   └── ComparisonTab.tsx (비교 분석)
   └── hooks/
       └── useReportData.ts (데이터 페칭)
   ```
   - 각 탭을 독립적인 유닛 테스트 대상으로 변경
   - undefined 방어 로직을 각 탭 내부로 격리

2. **InputSection.tsx (824줄) → 구성 컴포넌트 분리**:
   ```
   components/input/
   ├── InputSection.tsx (폼 컨테이너)
   ├── FileUploader.tsx
   ├── ParameterControl.tsx
   └── hooks/
       └── useInputValidation.ts
   ```

3. **E2E 테스트 기본 설정 표준화** (`e2e/fixtures.ts` 신규):
   ```typescript
   import { test as base, Page } from '@playwright/test';
   
   type TestFixtures = {
     authenticatedPage: Page;
   };
   
   export const test = base.extend<TestFixtures>({
     authenticatedPage: async ({ page }, use) => {
       // 로그인 로직
       await use(page);
     }
   });
   ```
   - 기존 `e2e/*.spec.ts` 파일들의 암묵적 any 제거

**구현 난이도**: High

**예상 임팩트**: High
- 테스트 커버리지 증가 (현재 E2E만 → 유닛 테스트 추가 가능)
- 개별 파일 변경 시 영향 범위 축소 → 릴리스 사이클 단축
- 타입 안정성 개선 → 런타임 에러 15-20% 감소 예상

**영향 파일**:
- `components/ReportTabs.tsx` (분해)
- `components/InputSection.tsx` (분해)
- `e2e/fixtures.ts` (신규)
- `e2e/pages.spec.ts`, `e2e/settings.spec.ts` (타입 정의 추가)
- `tsconfig.e2e.json` (설정 개선)

**예상 작업량**: 5-7일 (컴포넌트 분해 + 테스트 작성 + 통합)

## Idea 3: 스토리지 할당량 관리 시스템 도입으로 "localStorage quota exceeded" 반복 오류 제거

**발견 근거**:
- Git 로그에서 스토리지 관련 버그 반복:
  - `fix: prevent localStorage quota exceeded on flow analysis` (3ca6211)
  - `fix: restore image display using IndexedDB for large image storage` (46ba929)
- lib/storage.ts가 핫스팟 파일(4회 수정)
- 사용자 흐름: flow analysis → 이미지 저장 → localStorage 초과 → 분석 실패

**현재 문제**:
- localStorage는 도메인당 5-10MB 제한인데, 대용량 이미지/JSON을 제한 없이 저장 시도
- 오류 발생 시 사용자에게 명확한 가이드 없음 (자동 복구 불가)
- 스토리지 사용량 모니터링 기능 부재 → 예방 불가능
- 설정(history/page.tsx)에서 사용자가 수동으로 관리할 방법 없음

**제안**:
1. **스토리지 계층 재설계** (`lib/storage/storage-manager.ts` 신규):
   - localStorage 사용량 실시간 모니터링
   - 용량 초과 임계값(80%) 도달 시 자동 정책 실행:
     - 가장 오래된 분석 결과부터 삭제
     - 압축 알고리즘 적용 (이미지는 WebP로 변환)
   - 자동 복구 불가능 시 IndexedDB 폴백

2. **사용자 인터페이스 추가** (settings/page.tsx 확장):
   ```typescript
   // 설정 페이지에 "스토리지" 섹션 추가
   - 사용 중인 용량 표시 (MB)
   - "자동 정리" 토글 (기본값: ON)
   - "수동 정리" 버튼
   - "분석 결과 보관 기간" 설정 (7일/30일/무제한)
   ```

3. **모니터링 & 경고**:
   - localStorage 용량 80% 이상 시 `components/OnboardingBanner.tsx` 스타일로 배너 표시
   - "스토리지가 거의 가득 찼습니다. 오래된 결과를 정리하세요" 메시지

4. **API 레벨 응답 최적화** (app/api/analyze/route.ts):
   - 응답 크기 제한 추가 (예: 대용량 이미지 제외, 썸네일만 반환)

**구현 난이도**: Medium

**예상 임팩트**: Medium
- "localStorage quota exceeded" 에러 → 완전 제거
- 사용자 데이터 손실 없음 (자동 정책으로 우아한 처리)
- 반복되는 storage.ts 수정 → 향후 0회 예상
- 사용자 신뢰도 향상 (자동 관리, 수동 제어 모두 지원)

**영향 파일**:
- `lib/storage.ts` (리팩토링 + 용량 관리 로직 추가)
- `lib/storage/storage-manager.ts` (신규)
- `lib/storage/cleanup-policy.ts` (신규)
- `app/settings/page.tsx` (UI 추가)
- `components/OnboardingBanner.tsx` (경고 통합)
- `app/api/analyze/route.ts` (응답 최적화)

**예상 작업량**: 3-4일 (스토리지 로직 + UI + 테스트)
