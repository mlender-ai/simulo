### Idea 1: HeatmapViewer 좌표 파이프라인 교체 — OCR 의존 제거

**발견 근거**:
- `HeatmapViewer.tsx`가 7일 내 핫스팟 신규 진입, 관련 fix 커밋 3건 연속 발생
  - `e72e3cc` fix: improve heatmap zone accuracy and prevent overlap
  - `fe23647` feat: improve heatmap zone accuracy with grid-based prompt
  - `cb4ebe4` fix: restore panel, add 3-variant support + PNG save; improve heatmap accuracy via OCR coords
- OCR 좌표 추출 전용 API 엔드포인트(`app/api/ocr-extract/route.ts`)가 별도 존재하나, 좌표 정확도 문제가 반복 수정에도 해결되지 않음

**현재 문제**:
- 현재 구조는 스크린샷 이미지 → OCR 좌표 추출 → 클로드 그리드 프롬프트 → 히트맵 렌더링의 4단계 체인으로, 각 단계에서 오차가 누적됨
- OCR이 이미지 내 텍스트/요소 좌표를 정확히 추출하지 못하는 엣지케이스(아이콘-only 영역, 저해상도 캡처)에서 히트맵이 실제 UI 요소와 어긋나게 표시됨
- `HeatmapViewer.tsx`가 렌더링과 좌표 보정 로직을 모두 보유하여 수정마다 예상치 못한 부작용 발생

**제안**:
1. **좌표 기준점을 OCR에서 정규화 그리드로 교체**: 이미지를 고정 N×M 셀(예: 12×20)로 분할하고, 클로드에게 "어느 셀이 고관심 영역인가"를 셀 ID(row, col) 배열로 반환하도록 프롬프트를 재설계. OCR 파싱 단계를 완전 제거
2. **좌표 → 렌더링 변환을 순수 함수로 분리**: `lib/heatmap/gridToPixel.ts`를 신설하여 `(cellId, containerSize) → {x, y, w, h}` 변환 로직을 단위 테스트 가능한 형태로 추출
3. **`HeatmapViewer.tsx`는 렌더링만 담당**: 비즈니스 로직을 `lib/heatmap/`으로 이관하고 컴포넌트는 props 수신 및 시각화만 수행

```
lib/heatmap/
├── gridToPixel.ts     (셀ID → px 좌표 변환, 순수 함수)
├── gridPrompt.ts      (Claude용 그리드 프롬프트 생성)
├── types.ts           (HeatmapCell, GridConfig 등)
└── __tests__/
    └── gridToPixel.test.ts
```

**구현 난이도**: Medium
**예상 임팩트**: High
**영향 파일**:
- `components/HeatmapViewer.tsx`
- `app/api/ocr-extract/route.ts`
- `lib/prompts.ts`
- `lib/claude.ts`
**예상 작업량**: 2~3일

### Idea 2: ImprovementPanel 상태 머신 도입 — "버튼이 아무것도 안 하는" 버그 구조적 차단

**발견 근거**:
- 7일 내 `ImprovementPanel.tsx`와 generate-improvement 흐름에서 연쇄 fix 4건 발생:
  - `fd360ff` fix: fix reanalyze button doing nothing
  - `cb4ebe4` fix: restore panel, add 3-variant support + PNG save
  - `d0ec8cd` feat: replace improvement panel with screenshot panel (max 3 captures) — 패널 자체를 교체하는 근본적 리셋 시도
  - `9e12145` fix: rewrite generator prompt for surgical fixes, not full redesign
- 같은 기간 export 기능에서는 `11ccfcc`에서 "5단계 상태 머신"을 이미 도입하여 안정화에 성공한 패턴이 존재 — 동일 패턴 미적용 영역에서 버그가 집중됨

**현재 문제**:
- `ImprovementPanel.tsx`의 재분석 버튼, 생성 버튼, 저장 버튼이 각각의 로컬 `useState` boolean으로 관리되어 상태 조합 폭발(버튼 A 클릭 중 B 클릭, 응답 도착 전 컴포넌트 언마운트 등)을 처리하지 못함
- "버튼이 아무것도 안 한다"는 증상은 이전 async 요청이 진행 중임에도 상태가 idle로 보이는 경쟁 조건(race condition)에서 반복 발생하는 패턴임
- 프롬프트 재작성(9e12145)은 출력 품질 개선이지, 상태 관리 버그의 근본 해결이 아님

**제안**:
- Export Hub에서 검증된 5단계 상태 머신 패턴을 ImprovementPanel에 동일하게 적용
- `useImprovementMachine` 훅을 신설하여 상태를 명시적 열거형으로 관리:

```typescript
// lib/improvement/useImprovementMachine.ts
type ImprovementState =
  | { status: 'idle' }
  | { status: 'generating'; variantCount: number }
  | { status: 'ready'; variants: ImprovementVariant[] }
  | { status: 'saving'; variantIndex: number }
  | { status: 'error'; message: string };

// 허용된 전이만 실행 — 중복 클릭, 경쟁 조건 구조적 차단
function transition(state: ImprovementState, event: ImprovementEvent): ImprovementState
```

- `ImprovementPanel.tsx`는 상태 머신의 현재 state를 구독하여 렌더링만 수행
- 각 상태별 UI(로딩 스피너, 비활성 버튼, 에러 배너)를 상태 머신 전이에서 자동 파생

**구현 난이도**: Medium
**예상 임팩트**: High
**영향 파일**:
- `components/ImprovementPanel.tsx`
- `app/api/generate-improvement/route.ts`
- `lib/improvement/opusGenerator.ts` (신설: `lib/improvement/useImprovementMachine.ts`)
**예상 작업량**: 1.5~2일

### Idea 3: `lib/claude.ts` 분리와 빌드 FAIL 동시 해소 — 기술 부채 청산 패키지

**발견 근거**:
- `lib/claude.ts`(893줄)와 `lib/prompts.ts`가 각각 핫스팟 Top 5에 위치하며 7일간 4~5회 변경
- QA 리포트에서 TypeScript 빌드 FAIL이 2주 이상 지속 미해결 상태 (`@playwright/test` 모듈 미등재, `tsconfig.json` 미동기화)
- 두 문제가 동일한 작업 범위(타입 설정, 모듈 구조)에 걸쳐 있어 함께 해결할 때 효율이 높음
- 이전 아이디어 리포트(04-24, 04-25)에서 반복 제기됐으나 미실행 상태 — 빌드 FAIL이 지속되는 한 신규 기능의 타입 안전성 보장 불가

**현재 문제**:
- `lib/claude.ts` 893줄에 Claude API 통신, 프롬프트 조립, 응답 파싱, 에러 핸들링이 혼재하여 어느 파일을 수정해도 claude.ts 또는 prompts.ts가 함께 변경됨 → 핫스팟 고착화
- 빌드 FAIL 상태에서 CI가 실질적인 게이트 역할을 못하고 있어 타입 에러가 프로덕션까지 도달할 위험
- `e2e/settings.spec.ts`의 암묵적 `any` 타입 + `tsconfig.json` 미동기화로 E2E 테스트 IDE 지원 불가 → 테스트 작성 마찰

**제안**:
1. **빌드 FAIL 즉시 처리** (Day 1, 2시간):
```bash
npm install --save-dev @playwright/test@latest @types/node
```
```json
// tsconfig.json 추가
{
  "compilerOptions": {
    "types": ["@playwright/test", "node"],
    "moduleResolution": "node"
  }
}
```

2. **`lib/claude.ts` 기능별 분리** (Day 1~2):
```
lib/claude/
├── index.ts           (공개 API 재-export, 하위 호환)
├── client.ts          (Claude API 통신, 재시도 로직)
├── promptBuilder.ts   (프롬프트 조립 — lib/prompts.ts 흡수)
├── responseParser.ts  (응답 파싱, 스키마 검증)
└── types.ts           (ClaudeRequest, ClaudeResponse 등)
```

3. **E2E 픽스처 표준화** (Day 2, 기존 QA 리포트 제안 채택):
```typescript
// e2e/fixtures.ts
import { test as base, Page } from '@playwright/test';
export const test = base.extend<{ page: Page }>({});
// 모든 spec 파일이 이 fixtures에서 import
```

- `lib/claude/index.ts`에서 기존 import 경로(`lib/claude`)를 그대로 재-export하여 다른 파일 변경 없이 하위 호환 유지

**구현 난이도**: Medium
**예상 임팩트**: High
**영향 파일**:
- `lib/claude.ts` → `lib/claude/` (분리)
