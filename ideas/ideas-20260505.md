### Idea 1: 개선안 생성 파이프라인 — 상태 머신 기반 재설계

**발견 근거**:
- `ImprovementPanel.tsx`가 2주 연속 핫스팟 상위 유지 (이번 주 4회, 지난 주 5회)
- race condition fix가 동일 컴포넌트에서 2회 반복 (fd360ff, 41a89db) 후 이번 주 재분석 버튼 관련 fix(74b24c2)가 또 발생 — 패치 누적으로는 근본 해결 불가 패턴
- `lib/improvement/opusGenerator.ts`도 2회 수정, 프롬프트 재작성(9e12145, 20eb4f1)이 연속 발생해 생성 품질도 불안정

**현재 문제**:
재분석 버튼 클릭 → API 호출 → 결과 렌더링으로 이어지는 흐름이 컴포넌트 내부 `useState` + `useEffect` 조합으로 관리되고 있어, 비동기 타이밍 충돌(race condition)이 구조적으로 재발한다. 변형 수 선택(1–3), 참조 이미지 입력, 멀티 스크린 생성 등 입력 파라미터가 누적될수록 분기 복잡도가 선형 증가하는 구조.

**제안**:
`useReducer` 또는 경량 상태 머신(XState 또는 직접 구현)을 도입해 생성 파이프라인의 상태를 명시적 전이(idle → pending → streaming → done | error)로 표현한다. 상태 전이 외부에서는 API 호출이 불가능하도록 가드를 두어 race condition을 구조적으로 차단한다.

```
lib/improvement/
├── useImprovementMachine.ts   ← 상태 머신 훅 (신규)
├── opusGenerator.ts           ← 순수 API 호출 레이어 (기존 정리)
├── improvePromptBuilder.ts    ← 프롬프트 빌더 분리 (신규)
└── types.ts                   ← 공유 타입
components/
└── ImprovementPanel.tsx       ← 상태 머신 훅 소비만
```

`ImprovementPanel.tsx`는 상태 머신 훅을 소비하는 뷰 레이어로만 남기고, 비동기 오케스트레이션 로직을 완전히 분리한다.

**구현 난이도**: Medium
**예상 임팩트**: High
**영향 파일**:
- `components/ImprovementPanel.tsx`
- `lib/improvement/opusGenerator.ts`
- `lib/prompts.ts`
- `app/api/generate-improvement/route.ts`
**예상 작업량**: 2–3일

### Idea 2: 페이지 레이어 팽창 방지 — Dashboard / History 데이터 레이어 분리

**발견 근거**:
- 이번 주 핫스팟에 `app/page.tsx`(3회), `app/dashboard/page.tsx`(3회), `app/history/page.tsx`(2회), `app/settings/page.tsx`(2회)가 새롭게 진입 — 이전 주까지는 컴포넌트·lib 레이어가 핫스팟이었으나 페이지 직접 변경으로 확산
- `app/dashboard/page.tsx`에서 insights API key 전달 버그(16bc4d1), minimum data guard 추가(93526e0), score 해석·벤치마크 추가(0c0629b)가 모두 동일 페이지 파일에서 처리 — 데이터 페칭, 가드 로직, UI 렌더링이 단일 파일에 혼재
- `app/history/page.tsx`는 `useHistory` 훅 분리(f585a65)가 이미 진행됐으나 핫스팟 유지 — 훅 분리만으로는 불충분

**현재 문제**:
페이지 컴포넌트가 API 호출, 에러 가드, 통계 계산, 필터 상태, UI 렌더링을 모두 직접 담당하고 있어, 기능 추가 시 매번 페이지 파일이 변경되는 구조다. `app/dashboard/stats/route.ts`와 페이지 간 타입 계약(`typed stats contract`가 이번 주 추가됐음에도)이 느슨하게 유지되어 버그 발생 시 원인 추적이 어렵다.

**제안**:
Dashboard와 History 각각에 전용 데이터 훅을 도입하고, 페이지 컴포넌트는 레이아웃·렌더링만 담당하도록 계층을 명확히 분리한다.

```
lib/
├── hooks/
│   ├── useDashboardStats.ts   ← 통계 페칭 + 타입 계약 (신규)
│   ├── useHistory.ts          ← 기존 훅 강화
│   └── useInsights.ts         ← API key 전달 로직 캡슐화 (신규)
├── stats/
│   ├── statsCalculator.ts     ← 벤치마크·해석 로직 분리 (신규)
│   └── types.ts               ← dashboard stats 타입 단일 출처
app/
├── dashboard/page.tsx         ← useDashboardStats 소비만
└── history/page.tsx           ← useHistory 소비만
```

`app/api/dashboard/stats/route.ts`의 응답 스키마를 `lib/stats/types.ts`에서 단일 출처로 관리해 프론트-백 타입 불일치를 컴파일 타임에 잡는다.

**구현 난이도**: Medium
**예상 임팩트**: Medium
**영향 파일**:
- `app/dashboard/page.tsx`
- `app/history/page.tsx`
- `app/api/dashboard/stats/route.ts`
- `app/api/history/route.ts`
- `components/UsabilityReportTabs.tsx`
**예상 작업량**: 2일

### Idea 3: 리포트 → 팀 액션 연결 — Share & Next Step 워크플로우

**발견 근거**:
- 이번 주 리포트 페이지에 재분석 버튼(74b24c2), 이슈 탭 복사(590d8e4), tab URL deep-link(72b92a2), score delta 배지(3df6de8)가 집중 추가됨 — 사용자가 리포트 결과를 **외부로 가져가거나 팀과 공유하려는 수요**가 UX 요청 형태로 표면화되고 있는 신호
- 앱 구조에 `app/share/[id]/page.tsx`가 이미 존재하나, 관련 커밋이 최근 7일 내 전무 — 공유 기능이 존재하지만 리포트 UX 흐름과 연결이 끊긴 상태
- `components/ShareExportPanel.tsx`가 컴포넌트로 존재하지만 어떤 페이지에서 노출되는지 커밋 이력상 불명확

**현재 문제**:
분석 결과 확인(리포트) → 개선안 생성 → 팀 공유 → 후속 작업 지정으로 이어지는 자연스러운 UX 흐름이 단절되어 있다. 이슈 복사 버튼과 CSV 내보내기가 개별 기능으로 존재하지만 "이 이슈를 누가 처리할지", "언제까지 개선할지"를 리포트 내에서 지정할 수 없어 Simulo가 단순 분석 도구에 머무른다.

**제안**:
리포트 페이지에 **"팀 공유 + 액션 지정"** 패널을 추가해 분석 결과에서 실행 워크플로우까지 단일 흐름으로 연결한다.

1. **공유 링크 강화**: `app/share/[id]`를 리포트 뷰어로 활성화하고, 공유 시 특정 탭·이슈로 앵커 가능하도록 기존 `?tab=` deep-link 확장
2. **이슈별 액션 태깅**: 이슈 카드에 `우선순위 변경`, `담당자 메모`, `완료 체크` 인터랙션 추가 (로컬 상태, DB 저장 선택)
3. **리포트 요약 내보내기**: 현재 CSV 외에 Slack 메시지 포맷 또는 마크다운 요약 클립보드 복사 지원

```
app/share/[id]/page.tsx         ← 공유 전용 리포트 뷰어 활성화
app/report/[id]/page.tsx        ← 액션 패널 통합
components/ShareExportPanel.tsx ← 기존 컴포넌트 확장
components/IssueActionCard.tsx  ← 이슈별 액션 태깅 (신규)
app/api/report/route.ts         ← 액션 상태 저장 엔드포인트 확장
```

**구현 난이도**: Medium
**예상 임팩트**: High
**영향 파일**:
- `app/share/[id]/page.tsx`
- `app/report/[id]/page.tsx`
- `components/ShareExportPanel.tsx`
- `app/api/report/route.ts`
- `components/ReportTabs.tsx`
**예상 작업량**: 3–4일
