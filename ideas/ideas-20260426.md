### Idea 1: 멀티-입력 타입 통합 라우터 UI — "분석 시작" 단일 진입점

**발견 근거**:
- 최근 7일 신규 입력 채널 3종 추가: URL 라이브 분석(`feat: URL 라이브 분석 구현`), 영상 업로드(`feat: 영상 업로드 + 프레임 분석`), 코드→Figma 역방향(`feat: 코드 → Figma 역방향 분석`)
- 앱 구조상 `app/analyze/page.tsx`, `app/flow-builder/page.tsx`, `app/code-to-figma/page.tsx`가 **각각 독립 페이지**로 분산되어 있음
- `components/InputSection.tsx`가 4회 수정된 핫스팟 — 입력 유형별 분기 로직이 단일 파일에 누적 중 (824줄)

**현재 문제**:
- 사용자가 "이미지 분석"을 원하는지, "URL 분석"을 원하는지, "영상 분석"을 원하는지 어떤 페이지로 가야 하는지 불명확
- 입력 채널이 늘어날수록 네비게이션 depth 증가 → 첫 사용자의 온보딩 마찰 누적
- `InputSection.tsx`에 분기 로직이 모여 824줄까지 팽창 → 신규 입력 타입 추가 시 회귀 버그 위험

**제안**:
- `app/page.tsx`를 **"분석 시작" 허브 페이지**로 재설계: 입력 타입 카드 선택(이미지 / URL / 영상 / Figma / 코드) → 선택 후 해당 전용 페이지로 라우팅
- `InputSection.tsx`를 입력 타입별 하위 컴포넌트로 분해(`ImageInput.tsx`, `UrlInput.tsx`, `VideoInput.tsx`)하여 각 페이지에서 단독 사용
- 선택한 입력 타입을 `localStorage`에 기억해 재방문 시 마지막 타입으로 자동 포커스

```
components/input/
├── InputRouter.tsx        ← 타입 선택 허브 (신규)
├── ImageInput.tsx         ← 기존 로직 분리
├── UrlInput.tsx           ← url 핸들러 연동
├── VideoInput.tsx         ← upload-video 연동
└── hooks/
    └── useLastInputType.ts
```

**구현 난이도**: Medium
**예상 임팩트**: High
**영향 파일**:
- `app/page.tsx`
- `components/InputSection.tsx`
- `app/analyze/page.tsx`
- `app/api/analyze/handlers/url.ts`
- `app/api/analyze/handlers/image.ts`

**예상 작업량**: 3~4일

### Idea 2: 분석 핸들러 Registry 기반 — 입력→리포트 파이프라인 계약 검증 레이어

**발견 근거**:
- `app/api/analyze/registry.ts` 신설 + 핸들러 디렉토리(`handlers/figma.ts`, `url.ts`, `flow.ts`, `image.ts`, `comparison.ts`) 분리 완료 — 아키텍처는 갖춰졌으나 **각 핸들러의 출력 스키마 일관성을 검증하는 계층이 없음**
- QA 리포트: 타입 체크 FAIL 지속, `app/api/analyze/route.ts` 4회 수정(핫스팟 2위)
- `feat: E2E 타입 안전성 강화 + 입력 검증 스키마 분리` 커밋이 있었으나 핸들러 출력 측 스키마는 미포함

**현재 문제**:
- 5개 핸들러(image, url, figma, flow, comparison)가 각각 다른 형태의 JSON을 반환할 수 있으며, `report/[id]/page.tsx`에서 이를 **방어적으로 처리하는 표준이 없음**
- 새로운 입력 타입 추가 시 리포트 렌더링 측에서 undefined 접근 에러 발생 가능 (이전 주 JSON 파싱 버그의 재발 경로와 동일)
- `app/report/[id]/page.tsx`가 핫스팟 3위(3회 수정) — 핸들러 출력 변경 때마다 리포트 페이지도 연쇄 수정 필요

**제안**:
- `app/api/analyze/handlers/types.ts`에 **핸들러 출력 공통 스키마(Zod)** 정의:

```typescript
// handlers/types.ts
import { z } from 'zod';

export const BaseAnalysisOutputSchema = z.object({
  handlerType: z.enum(['image', 'url', 'video', 'figma', 'flow', 'comparison']),
  issues: z.array(IssueSchema),
  summary: z.string(),
  heatmapZones: z.array(HeatmapZoneSchema).optional(),
  metadata: z.record(z.unknown()).optional(),
});

export type AnalysisOutput = z.infer<typeof BaseAnalysisOutputSchema>;
```

- `registry.ts`에서 각 핸들러 등록 시 출력 스키마 통과 여부를 **자동 검증(pre-flight validation)** 후 저장
- 검증 실패 시 구조화된 에러(`{ code: 'SCHEMA_MISMATCH', handler, field }`)를 반환하여 리포트 페이지가 안전하게 fallback 렌더링 가능하게

```
app/api/analyze/
├── registry.ts           ← 핸들러 등록 + 출력 검증 추가
├── handlers/
│   ├── types.ts          ← 공통 출력 스키마 (Zod) 확장
│   ├── image.ts          ← 스키마 준수 강제
│   ├── url.ts
│   ├── video.ts          ← 신규 핸들러
│   └── ...
```

**구현 난이도**: Medium
**예상 임팩트**: High
**영향 파일**:
- `app/api/analyze/registry.ts`
- `app/api/analyze/handlers/types.ts`
- `app/api/analyze/handlers/image.ts`
- `app/api/analyze/handlers/url.ts`
- `app/api/analyze/handlers/figma.ts`
- `app/api/analyze/handlers/flow.ts`
- `app/api/analyze/route.ts`
- `app/report/[id]/page.tsx`

**예상 작업량**: 2~3일

### Idea 3: 히스토리 페이지 — 입력 타입별 필터링 + 분석 재실행 단축키

**발견 근거**:
- `app/history/route.ts` 존재, `app/history/page.tsx` 존재하나 최근 7일 **단 한 번도 수정되지 않음** — 기능 확장 없이 정체 중
- `feat: 이슈 탭 화면별 필터링 및 썸네일 표시 개선` 커밋: 리포트 내부 필터링은 개선됐으나 히스토리 레벨 필터링은 미적용
- 입력 채널이 5종(이미지/URL/영상/Figma/코드)으로 확장되었으나 히스토리에서 타입 구분 없이 나열될 경우 **사용자가 과거 분석을 재활용하기 어려움**
- `fix: 히스토리에서 PNG 버튼 제거` 커밋 — 히스토리 페이지의 액션 UI가 아직 정비 중임을 시사

**현재 문제**:
- 히스토리 목록에서 "이 분석이 URL 분석인지, 이미지 분석인지" 구분할 수 없어 원하는 분석 결과를 찾기 위해 하나씩 열어봐야 함
- 유사한 화면을 반복 테스트하는 사용자(QA/디자이너)가 **"이 조건으로 다시 분석"** 을 하려면 입력값을 처음부터 다시 입력해야 함
- localStorage 환경에서 히스토리 데이터가 쌓일수록 `StorageWarningBanner` 노출 빈도 증가 → 히스토리 관리 도구 필요성 증가

**제안**:
- 히스토리 목록에 **입력 타입 배지** (🖼 이미지 / 🔗 URL / 🎬 영상 / 🎨 Figma / 💻 코드) 및 타입별 필터 탭 추가
- 각 히스토리 카드에 **"동일 조건으로 재분석"** 버튼 추가 — 저장된 입력 파라미터(타깃 유저, 분석 유형, URL 등)를 복원하여 분석 페이지로 이동
- **히스토리 일괄 삭제** 기능 추가 (타입별/날짜별 선택 삭제) — `StorageWarningBanner` 발생 시 히스토리 페이지 직접 링크로 안내

```
app/history/
└── page.tsx              ← 필터 탭 + 재분석 버튼 + 일괄 삭제 UI 추가

components/
├── HistoryCard.tsx        ← 신규: 타입 배지 + 재분석 CTA
└── HistoryFilterBar.tsx   ← 신규: 입력 타입 필터

lib/
└── storage.ts             ← 히스토리 타입 필드 추가, 일괄 삭제 함수
```

**구현 난이도**: Low
**예상 임팩트**: Medium
**영향 파일**:
- `app/history/page.tsx`
- `app/api/history/route.ts`
- `lib/storage.ts`
- `components/StorageWarningBanner.tsx`

**예상 작업량**: 1~2일
