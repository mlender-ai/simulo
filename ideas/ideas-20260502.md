### Idea 1: AI 개선안 생성 파이프라인 계약 레이어 도입 (ImprovementContract)

**발견 근거**:
- `ImprovementPanel.tsx` 최근 7일 5회 수정으로 핫스팟 1위
- `lib/prompts.ts`·`lib/claude.ts`·`lib/improvement/opusGenerator.ts` 동시 핫스팟 진입
- `reanalyze` 버튼 동일 버그 2회 재발(fd360ff → 41a89db), 프롬프트 전면 재작성(9e12145) 병행
- 이전 아이디어(2026-04-29 Idea 1)에서 HeatmapContract 계층 도입을 제안했으나, 당시는 히트맵→AI 방향이었고 본 아이디어는 **AI 출력→UI 렌더링** 역방향 계약에 집중

**현재 문제**:
- `opusGenerator.ts`가 생성하는 개선안 데이터 구조와 `ImprovementPanel.tsx`가 소비하는 구조 사이에 명시적 타입 계약이 없음
- 프롬프트 변경 시(`lib/prompts.ts`) AI 출력 형태가 달라져도 패널 컴포넌트가 이를 감지하지 못하고 silent failure 발생 → `reanalyze` 버튼 무반응처럼 UX 단에서야 증상이 드러남
- `lib/claude.ts`(893줄)가 API 통신·프롬프트 조립·응답 파싱을 모두 담당해 어느 단계에서 실패했는지 추적 불가

**제안**:
1. `lib/improvement/types.ts` 신규 생성 — `ImprovementOutput` 인터페이스를 Zod 스키마로 정의하고 `opusGenerator.ts` 출력과 `ImprovementPanel` props를 동일 타입으로 고정
2. `lib/claude.ts` → `lib/claude/` 디렉토리 분리: `ai-client.ts`(API 통신), `response-parser.ts`(출력 파싱 + Zod 검증), `prompt-builder.ts`(프롬프트 조립)로 단일 책임 분리
3. `response-parser.ts`에서 스키마 검증 실패 시 `ImprovementPanel`에 `errorState`를 명시적으로 전달하는 에러 경계 추가 — silent failure 제거

```
lib/
├── claude/
│   ├── ai-client.ts        # Claude API fetch
│   ├── prompt-builder.ts   # 프롬프트 조립
│   ├── response-parser.ts  # Zod 검증 + 에러 경계
│   └── types.ts
├── improvement/
│   ├── opusGenerator.ts
│   └── types.ts            # ImprovementOutput (신규)
```

### Idea 2: TypeScript 빌드 게이트 — CI 블로커 자동화 및 타입 안전망 복구

**발견 근거**:
- QA 리포트: 빌드 ⚠️ FAIL, 타입 체크 ❌ FAIL 동시 발생
- `@playwright/test` 모듈 미등록, `e2e/settings.spec.ts` 암묵적 `any` 3건
- `package-lock.json` 584줄 증가 — 의존성 설치는 됐으나 tsconfig와 미동기화
- 기능 커밋(38개 중 25개)이 빌드 실패 상태에서 계속 머지된 정황 — 타입 오류가 기능 개발 속도를 따라가지 못함

**현재 문제**:
- 빌드와 타입 체크가 분리 실패인데도 E2E가 PASS로 나오는 모순 → CI가 타입 오류를 블로킹하지 않는 구조
- `tsconfig.json`이 Playwright 설정과 미동기화된 채로 방치되어 로컬/CI 환경 결과가 달라지는 "작동하는 것처럼 보이는" 상태 지속
- 893줄 `claude.ts`, 824줄 `InputSection.tsx` 등 대형 파일에 타입 any가 누적되면 추후 리팩토링 비용이 기하급수적으로 증가

**제안**:
1. **즉시 수정 (1단계)**: `npm install --save-dev @playwright/test@latest @types/node` 실행 후 `tsconfig.json`에 `"types": ["@playwright/test", "node"]` 추가, `e2e/settings.spec.ts` 3개 암묵적 any 명시 타입으로 교체
2. **CI 게이트 강화 (2단계)**: GitHub Actions(또는 Vercel 빌드 훅)에 `tsc --noEmit` 스텝을 별도 job으로 추가해 타입 오류 시 머지 블로킹
3. **회귀 방지 (3단계)**: `e2e/fixtures.ts` 공통 픽스처 파일 도입으로 `Page` 타입 중앙화, `tsconfig.strict: true` 단계적 적용(신규 파일부터 먼저 적용)

```jsonc
// tsconfig.json 추가
{
  "compilerOptions": {
    "skipLibCheck": true,
    "moduleResolution": "node",
    "types": ["@playwright/test", "node"]
  }
}
```

```yaml
# .github/workflows/ci.yml 추가 스텝
- name: Type Check
  run: npx tsc --noEmit
  # 실패 시 머지 블로킹
```

**구현 난이도**: Low
**예상 임팩트**: High
**영향 파일**:
- `tsconfig.json`
- `playwright.config.ts`
- `e2e/settings.spec.ts`
- `package.json`
- `.github/workflows/ci.yml` (신규)
- `e2e/fixtures.ts` (신규)

**예상 작업량**: 0.5~1일

### Idea 3: 프로덕트 모드(야핏무브 ↔ 일반) 분석 설정의 URL 퍼시스턴스 및 공유 연동

**발견 근거**:
- `feat: product mode toggle(야핏무브 ↔ 일반)` (1011235) → `feat: general mode UI` (99f89ec) → `feat: domain-specific analysis focus areas` (c867763) 3개 커밋이 연속으로 모드 관련 기능을 빠르게 쌓아 올림
- `feat(report): tab URL deep-link support via ?tab= query param` (72b92a2)가 같은 주에 별도로 추가됨 — 탭은 딥링크를 지원하지만 모드/도메인 설정은 URL에 반영되지 않는 불일치 상태
- `app/share/[id]/page.tsx` 존재 — 공유 기능이 있으나 분석 컨텍스트(어떤 모드·도메인으로 분석했는지)가 공유 URL에 포함되지 않으면 수신자가 다른 맥락으로 리포트를 보게 됨

**현재 문제**:
- 사용자가 "일반 서비스 + 헬스케어 도메인" 설정으로 분석한 리포트를 팀원에게 공유해도, 수신자가 열면 기본 모드(야핏무브)로 렌더링되어 perspective 섹션이 다르게 노출됨
- 모드 설정이 새로고침 시 초기화되는지, localStorage에 유지되는지 코드 레벨에서 명시적 계약이 없어 UX가 예측 불가능한 상태
- `?tab=` 딥링크와 모드 파라미터가 별도 구현이면 URL 파라미터 관리 코드가 컴포넌트마다 파편화될 위험

**제안**:
1. **URL 퍼시스턴스**: 분석 실행 시 `?mode=general&domain=healthcare` 형태로 URL에 반영, Next.js `useSearchParams`·`useRouter` 기반으로 기존 `?tab=` 딥링크 구현과 통일된 방식으로 처리
2. **공유 URL 컨텍스트 포함**: `app/share/[id]/page.tsx`에서 저장된 모드·도메인 메타데이터를 리포트 헤더에 배지(Badge) 형태로 표시 — 수신자가 "어떤 조건으로 분석된 리포트인지" 즉시 인지
3. **중앙 쿼리 파라미터 훅**: `hooks/useAnalysisParams.ts` 신규 생성으로 `mode`, `domain`, `tab` 파라미터를 단일 훅에서 관리, 컴포넌트 파편화 방지

```typescript
// hooks/useAnalysisParams.ts (신규)
export function useAnalysisParams() {
  const searchParams = useSearchParams();
  return {
    mode: searchParams.get('mode') ?? 'yafit',        // 'yafit' | 'general'
    domain: searchParams.get('domain') ?? 'general',  // 도메인 슬러그
    tab: searchParams.get('tab') ?? 'overview',
  };
}
```

**구현 난이도**: Medium
**예상 임팩트**: Medium
**영향 파일**:
- `app/page.tsx`
- `app/analyze/page.tsx`
- `app/share/[id]/page.tsx`
- `app/report/[id]/page.tsx`
- `components/InputSection.tsx`
- `hooks/useAnalysisParams.ts` (신규)

**예상 작업량**: 1~2일
