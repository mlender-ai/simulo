> "요구사항을 아무리 잘 써도, AI가 참조하는 파일에 쓰레기가 섞여 있으면 결과도 오염된다."
> "불안함을 인간 개입으로 해결하지 말고, 정확한 코드를 생성하고 강제하는 체계를 만들어라."

## 왜 하네스 엔지니어링인가

```
복잡한 작업을 에이전트에게 맡기면 품질이 떨어지는 이유:

  컨텍스트 한계 (200K 토큰 = 큰 작업에 부족)
  컨텍스트 오염 (무관/잘못된 정보가 섞임)
  단일 에이전트의 맥락 과부하

  하네스 엔지니어링으로 해결:
     역할이 분리된 TF처럼 운영
     큰 작업은 잘게 쪼개서 위임
     구현 담당 + 검증 담당이 협력
     컨텍스트는 핵심만, 깨끗하게
```

## 컨텍스트 오염 — 데이터가 말한다

```
오염 종류별 정확도 하락 (출처: research.trychroma.com):

  무관한 정보 포함  →  정확도 약 10% 하락
  잘못된 정보 포함  →  정확도 30%+ 하락
  복잡한 작업 시    →  최대 70%까지 하락
```

**결론:** 프롬프트 품질보다 컨텍스트 청결도가 결과 품질을 더 크게 좌우한다.

## 컨텍스트 오염 방지 3원칙

---

### 원칙 1 — 폴더 구조 룰 문서화 (지도 제공)

AI는 코드베이스 탐색 시 **모든 파일을 읽지 않는다**. 관련 있어 보이는 파일을 검색해서 찾아가는데, 이 탐색이 부정확하면 엉뚱한 파일이 컨텍스트를 오염시킨다.

**해결책:** 프로젝트 루트에 AGENTS.md로 지도를 제공한다.

```markdown
# AGENTS.md — 폴더 구조 가이드

## 디렉토리 역할

src/
├── components/     # 재사용 UI 컴포넌트. pages에서 import해 사용
│   ├── common/     # 버튼, 인풋 등 범용 컴포넌트
│   └── domain/     # 비즈니스 도메인별 컴포넌트
├── pages/          # 페이지. components를 조합
├── hooks/          # 커스텀 React 훅. 비즈니스 로직 분리
├── services/       # 외부 API 통신 레이어
├── stores/         # 전역 상태 (Zustand)
├── types/          # TypeScript 타입 정의 (공유)
├── utils/          # 순수 함수 유틸리티
└── lib/            # 외부 라이브러리 설정/초기화

## 파일 간 관계
- pages → components → hooks → services
- 역방향 import 금지
- 도메인 컴포넌트는 common 컴포넌트에 의존 가능, 역방향 불가

## 작업별 참조 파일 힌트
- UI 수정: components/ + pages/
- API 연동: services/ + types/
- 상태 관리: stores/ + hooks/
- 공통 로직: utils/ + hooks/
```

> 핵심은 폴더 목록 나열이 아니라 **각 폴더의 역할과 파일 간 관계**를 명시하는 것.

---

### 원칙 2 — 코드 상의 정확한 용어 사용 (검색 정확도 향상)

AI는 내부적으로 **문자열 검색**으로 파일을 탐색한다. 프롬프트에 쓴 단어로 관련 파일을 찾는다.

```
잘못된 프롬프트:
  "버튼 클릭 시 동작을 변경해줘"
  → AI가 "버튼", "클릭", "동작" 같은 일반 단어로 검색
  → 관련 없는 파일 다수 참조 → 컨텍스트 오염

올바른 프롬프트:
  "SubmitButton 컴포넌트의 onClick 핸들러를 수정해줘"
  → 정확한 식별자로 탐색 → 해당 파일만 정확히 참조
  → 속도도 빠르고 오염도 없음
```

**규칙: 프롬프트에 반드시 포함할 것**

```
✅ 변수명, 함수명, 클래스명 정확히 명시
✅ 파일 경로나 파일명 포함
✅ 컴포넌트/모듈명을 코드 그대로 사용
❌ "그 버튼", "저 기능", "로그인 관련" 같은 모호한 표현
```

---

### 원칙 3 — 미사용 코드 및 구버전 코드 삭제 (쓰레기 제거)

AI는 주석 처리된 코드, 미사용 파일, 구버전 패턴을 **유효한 참고 자료**로 인식한다.

```
위험 상황:
  - 주석 처리된 옛날 구현 코드
  - deprecated된 함수/컴포넌트 (삭제 안 한 것)
  - 라이브러리 메이저 업데이트 후 구버전 패턴이 남은 코드
  - 실험적으로 만들었다가 안 쓰는 파일

결과:
  → AI가 구버전 API 패턴을 따라 코드 생성
  → 더 이상 존재하지 않는 함수 호출
  → 빌드 실패 or 런타임 에러
```

**"나중에 쓸 수 있으니 남겨두자" → 절대 금지. Git history에 있다. 지금 지워라.**

## 하네스 3대 기능

### 1. 스킬 (Skill) — 동적 지침 주입

작업 시 참고해야 할 지침, 라이브러리 사용법, 주의사항을 저장해두는 기능. 상황에 따라 동적으로 불러와지는 프롬프트.

```
용도:
  - 프로젝트에서 사용하는 API 스펙
  - 코드 컨벤션 규칙
  - 특정 라이브러리 사용 패턴

작동 방식:
  관련 작업 감지 → 해당 스킬 자동 참조 → 컨텍스트에 주입

위치: .claude/skills/[skill-name]/SKILL.md
```

### 2. 서브 에이전트 (Sub-agent) — 컨텍스트 분리 위임

특정 작업을 하위 에이전트에게 위임. 별도 세션에서 처리 후 **결과만** 메인 세션에 전달하고 소멸.

```
효과:
  메인 세션 컨텍스트 거의 소모하지 않음
  독립된 시각으로 작업 수행 (구현 맥락 없이 리뷰 가능)

적합한 작업:
  ✅ 코드 리뷰 (제3자 시선 필요)
  ✅ 보안 감사
  ✅ 문서 생성
  ✅ RAG 방식의 관련 ADR 추출

부적합한 작업:
  ❌ 맥락 전달이 복잡한 고관여 작업
     ("설명하느니 차라리 내가 하는 게 낫겠다" 싶은 작업)
     → 이런 경우는 포크 사용
```

### 3. 포크 (Fork) — 컨텍스트 복제 분기

현재 세션의 컨텍스트를 **그대로 복제**해 새 분기를 만든다. 이전 맥락을 재설명할 필요 없이 즉시 작업 착수.

```
적합한 작업:
  ✅ 설계 논의 완료 후 → 설계 의도 문서 작성 (포크)
  ✅ 구현 완료 후 → 코드 리뷰 반영 검토 (포크)
  ✅ 리서치 완료 후 → UX 세부 논의 (포크)
  ✅ 메인 세션 컨텍스트를 보존하며 부수 작업 처리

핵심 원칙:
  메인 세션 = 가장 깊은 이해 상태
  → 리뷰 반영 등 핵심 작업은 메인 세션에서만
  → 부수적 작업(문서화, QA 등)은 포크해서 처리
```

## 코드 리뷰 자동화 파이프라인

```
설계 논의 (메인 세션)
       │
       ├──[포크]──→ 설계 의도 문서 작성
       │              (design-intent.md)
       ▼
구현 (메인 세션)
       │
       ├──[서브에이전트]──→ 평가 기준 문서 생성
       │   관련 ADR 추출        (ADR + 코드 컨벤션 조합)
       │
       ├──[서브에이전트]──→ 코드 리뷰 실행
       │   (design-intent.md + 평가 기준 참조)
       │
       ▼
리뷰 반영 (메인 세션 보존)
       │
       ├──[포크]──→ PR 명세 / 작업 문서화
       ├──[포크]──→ QA 테스트
       │
       ▼
완료
```

**목표:** 구현 뒷단의 리뷰 과정에서 사람의 개입을 완전히 제거. 사람은 앞단 설계 논의에만 집중.

**신뢰 기준:** AI 리뷰와 사람 리뷰를 구분하지 못할 때까지 파이프라인 튜닝.

## 평가 기준 문서 2종

AI 코드 리뷰가 부족한 이유 = **두 가지 맥락이 빠져 있기 때문**

### 문서 1: code-convention.md (코드 컨벤션)

조직의 암묵적 코드 스타일을 명문화. 프레임워크 표준은 제외, **조직 고유 규칙만** 기록.

```markdown
# code-convention.md

## 네이밍 규칙
- 컴포넌트: PascalCase (UserProfileCard)
- 훅: use 접두사 + camelCase (useAuthState)
- 이벤트 핸들러: handle 접두사 (handleSubmit, handleClose)
- 상수: SCREAMING_SNAKE_CASE (MAX_RETRY_COUNT)
- 불리언 변수: is/has/can 접두사 (isLoading, hasError)

## 에러 처리
- try/catch에서 에러를 무시하지 않는다
- 사용자에게 보여줄 에러와 로깅용 에러를 구분한다
- API 에러는 services 레이어에서만 처리한다

## 컴포넌트 구조
- 파일당 하나의 컴포넌트 (기본 원칙)
- props는 interface로 정의, type 혼용 금지
- 200줄 초과 시 분리 검토

## import 순서
1. React/Next.js 기본
2. 외부 라이브러리
3. 내부 모듈 (@/components, @/hooks 등)
4. 타입 import (type 키워드 사용)
5. 스타일 파일

## 금지 패턴
- any 타입 사용 금지
- console.log 프로덕션 코드에 잔존 금지
- 중첩 3단계 이상 if/else 금지 (early return 사용)
- 직접 DOM 조작 금지 (React 상태 사용)

## 테스트
- 비즈니스 로직은 훅 레벨에서 테스트
- UI 컴포넌트는 사용자 인터랙션 기준으로 테스트
- Mock은 외부 API, DB에만 적용
```

### 문서 2: ADR (Architecture Decision Record)

"우리 조직은 어떤 설계를 지향하는가"를 담은 문서. **왜 이 설계를 선택했는가**가 핵심.

```markdown
# ADR/adr-001-state-management.md

## 제목
전역 상태 관리 라이브러리: Zustand 선택

## 날짜
2026-01-15

## 상태
Accepted

## 컨텍스트
사용자 인증 상태, 장바구니, UI 상태를 여러 컴포넌트에서 공유해야 함.
Redux, Jotai, Zustand를 검토함.

## 결정
Zustand 채택

## 이유
- 보일러플레이트 최소화 (Redux 대비 코드량 70% 감소)
- TypeScript 지원 우수, 별도 타입 선언 불필요
- 현재 팀 규모(3인)에서 Redux의 엄격한 구조는 오버엔지니어링
- 필요 시 Redux DevTools 연동 가능

## 트레이드오프
- 매우 복잡한 상태 로직에서는 Redux가 더 추적 용이
- 대규모 팀에서는 컨벤션 없이 오용 가능 → 사용 가이드 필수

## 결과
전역 상태는 stores/ 디렉토리에서만 정의.
로컬 UI 상태는 useState, 서버 상태는 TanStack Query 사용.
```

### ADR 서브에이전트 활용 (RAG 패턴)

ADR이 많아지면 관련 없는 내용이 컨텍스트를 오염시킴. 서브에이전트가 필터링.

```
[서브에이전트] ADR 추출기
─────────────────────────
1. 현재 작업 설계를 분석
2. 관련 ADR 항목만 추출
3. code-convention.md와 합쳐
4. 해당 작업 특화된 평가 기준 문서 생성
→ 구현/리뷰 에이전트에 전달
```

```markdown
# .claude/agents/adr-extractor.md

---
name: adr-extractor
description: >
  작업 설계를 분석하여 관련 ADR만 추출하고
  코드 컨벤션과 합쳐 작업별 평가 기준 문서를 생성한다.
  코드 리뷰 전 자동 실행.
tools: [Read, Glob, Write]
model: haiku
---

당신은 아키텍처 문서 큐레이터입니다.

입력: 현재 작업 설명 또는 PR diff
출력: 해당 작업에 관련된 ADR + 코드 컨벤션 = 평가 기준 문서

실행 순서:
1. ADR/ 디렉토리의 모든 ADR 파일 목록 파악
2. 현재 작업과 관련된 ADR만 선별 (무관한 ADR 절대 포함 금지)
3. code-convention.md에서 관련 섹션 추출
4. 합쳐서 .claude/memory/current-review-criteria.md 생성

출력 형식:
## 이 작업의 평가 기준

### 관련 아키텍처 결정
[선별된 ADR 요약]

### 적용 컨벤션
[관련 코드 컨벤션]

### 이 작업에서 특히 확인할 것
[위 내용에서 파생된 체크포인트]
```

```
언제:  LLM과 설계를 충분히 논의한 뒤
방법:  현재 세션을 포크 → 별도 세션에서 문서 작성

## 기능명
[구현 기능]

## 요구사항 요약
[무엇을 만들어야 하는가]

- Option B 검토: [왜 선택하지 않았는가]

[무엇을 희생했고 무엇을 얻었는가]

## 리뷰어가 주의해야 할 것
[이 설계를 모르면 오해할 수 있는 코드 패턴]

## 향후 변경 시 고려사항

## 세션 관리 핵심 규칙

```
메인 세션 = 가장 깊은 이해 상태의 컨텍스트
            → 절대 낭비하지 않는다

┌─────────────────────────────────────────────────┐
│ 메인 세션에서 할 것       │ 포크에서 할 것        │
├──────────────────────────┼────────────────────────┤
│ 설계 논의                │ 설계 의도 문서화        │
│ 핵심 구현                │ PR 명세 작성            │
│ 리뷰 반영 최종 판단      │ QA 테스트              │
│                          │ 작업 문서화             │
│                          │ 코드 리뷰 (서브에이전트) │
└──────────────────────────┴────────────────────────┘

절대 하지 말 것:
  리뷰 반영 시 새 세션 생성
  → 맥락 유실 → 리뷰에 대한 정확한 판단 불가

이상적인 컨텍스트 소모:
  구현 작업 완료 시 약 40% 소모가 적정
  (50% 초과 시 포크/서브에이전트 고려)

## 하네스 엔지니어링 체크리스트

### 프로젝트 초기 설정

```
□ AGENTS.md에 폴더 구조와 파일 간 관계 문서화
□ code-convention.md에 조직 고유 규칙 작성
□ ADR/ 디렉토리 생성 + 첫 번째 결정 기록
□ adr-extractor 서브에이전트 설정
□ code-reviewer 서브에이전트 설정
□ 미사용 파일 / 주석 처리된 코드 전수 삭제
```

### 기능 개발 전

```
□ 프롬프트에 정확한 식별자(함수명, 컴포넌트명) 사용
□ 설계 논의를 메인 세션에서 충분히 진행
□ 설계 확정 후 포크 → design-intent.md 작성
□ adr-extractor로 관련 ADR 추출 → 평가 기준 준비
```

### 구현 중

```
□ 메인 세션 컨텍스트 40% 이하 유지 목표
□ 부수 작업(문서화, QA)은 반드시 포크에서 처리
□ 메인 세션은 핵심 구현과 리뷰 반영용으로 보존
```

### 구현 완료 후

```
□ 서브에이전트로 코드 리뷰 실행
    (design-intent.md + 평가 기준 참조)
□ 리뷰 반영은 메인 세션에서 (새 세션 절대 금지)
□ PR 명세는 포크에서 작성
□ 이번 작업에서 새로운 패턴 발견 시 ADR 추가
□ 더 이상 쓰지 않는 코드 즉시 삭제
```

## 실전 팁

```
1. 구현 완료 시 컨텍스트 약 40% 소모가 이상적
2. 메인 세션은 티켓(기능) 단위로 생성하고 관리
3. 사람의 개입이 필요한 부분 = 앞단 설계 논의
4. 사람의 개입이 불필요한 부분 = 구현 + 리뷰 반복
5. AI 리뷰 신뢰 기준: 사람 리뷰와 구분 못 할 때까지 튜닝
```
